# Walker 当前网站设计调研报告

调研对象：本地站点 `http://127.0.0.1:4321/`  
调研时间：2026-05-03  
观察视角：桌面端与 390px 移动端  
重点观察页面：首页 `/`、航海日志 `/log`、文章页 `/log/articles/ai-design-workflow`、概念图谱 `/concept`、概念详情 `/concept/subtraction`、提示词库 `/prompts`、五层逻辑 `/framework`

## 1. 网站整体定位

Walker 当前站点的第一印象非常明确：它不是一个普通个人博客，而是一个带有强叙事隐喻的个人知识空间原型。首页首屏使用夜海、纸船、金色光路作为主视觉，配合 “行过万里水路，记录每一次停靠” 的句子，立刻把访问者带入“航行、停靠、探索、积累”的语境。

它更像以下几类形态的混合：

- 个人品牌主页：`Walker` 作为首屏绝对中心，视觉人格很强。
- 数字花园：概念图谱、关联概念、日志与提示词之间已经形成节点关系。
- 知识库：`/concept` 和 `/prompts` 已经不只是文章列表，而是面向长期沉淀的结构层。
- AI 时代工作台雏形：提示词卡片、复制按钮、与 AI 对话类内容，已经把“知识如何被重新调用”纳入系统。

目标访问者可能包括：

- 想了解 Walker 思维方式的新访客。
- 对 AI、设计、哲学、个人工作流感兴趣的同行。
- 回访的读者，希望继续从日志、概念或提示词进入。
- 未来的 AI agent，需要读取概念、内容、关系和可执行提示词。

网站传达的人格气质是：克制、内省、工程化、有航行感，带一点“夜里独自整理世界”的气质。它不像传统作品集那样展示成就，也不像纯 wiki 那样中性，而是把个人世界观作为内容组织的中心。

## 2. 信息架构

当前主导航是：

- `港口`：首页，承担品牌入口和总览。
- `海域`：日志列表，承载时间流内容。
- `概念`：概念图谱与五层逻辑入口。
- `提示词`：可复制的意图模块和工作流片段。
- 搜索：桌面端显示“搜索 + ⌘K”，移动端为图标。

内容分类使用三套体系叠加：

- 时间：日志按 `2026年4月下旬 / 中旬` 这样的时间段组织。
- 类型：文章、随想、对话、食谱、项目、视频、音频、图集、提示词等。
- 语义：每篇内容可绑定 `concepts`，概念自身还有 `related` 和 `layer`。

这是很好的底层结构。它已经不是“文章堆叠”，而是“内容作为概念证据”的模型。

新访客的问题在于：导航命名很有风格，但学习成本偏高。`港口 / 海域 / 概念 / 提示词` 有一致隐喻，却没有直接说明“海域就是全部日志”“概念是知识图谱”。对于第一次进入的人，首页首屏很美，但需要滚动后才知道这里能做什么。

老访客的继续探索路径比较清楚：可以从日志筛选、概念页、搜索或提示词库进入。但目前还缺少“上次更新”“最近扩展的概念”“某个概念最近新增了哪些内容”这类持续回访入口。

## 3. 首页设计

首页承担三件事：

- 建立 Walker 的视觉身份。
- 给出知识空间的核心隐喻。
- 把访客引向概念图谱、五层逻辑和最近内容。

首屏传达的是“夜海航行”的品牌意象。桌面端纸船、金色光路和 `Walker` 文本形成明确视觉中心，氛围成熟，辨识度高。移动端也保留了氛围，但纸船在 390px 宽度下被裁到左下角，只露出局部，品牌意象仍可识别，但“船”这个主体不够完整。

首页第二屏的概念图谱是全站最重要的结构信号。它告诉访问者：这里的内容不是线性博客，而是由概念连接起来的知识网络。五层逻辑则进一步把“人类主权 → 熵减命令 → 多元世界观 → 量变质变 → 做减法”组织成可读路径。

首页的问题是：首屏很强，但“下一步做什么”的引导不够直接。当前需要用户自行滚动，才能发现概念图谱和最近痕迹。对于想快速理解网站的人，可以增加一个低干扰的首屏入口，例如“进入概念图谱”“阅读最近日志”“打开提示词库”，或在首屏底部露出下一屏标题。

## 4. 视觉语言

### 色彩系统

主色非常稳定：

- 深海蓝 / 靛蓝：作为页面背景，形成夜海感。
- 金色：用于当前导航、概念符号、重要节点、提示词标签和强调状态。
- 羊皮纸白：用于标题和正文，带来温度。
- 灰蓝 / mist：用于辅助信息、日期、摘要和未激活状态。

这套配色高度贴合“行船、夜海、知识图谱、克制但有深度”的方向。问题是部分辅助文本使用 `text-parchment/30`、`text-mist/40` 这类低透明度，在移动端和深色背景上可读性偏弱。

### 字体选择

字体组合是：

- `Sora`：用于英文品牌、标题和导航。
- `Noto Sans SC`：承载中文正文。
- `JetBrains Mono`：用于日期、代码和技术感信息。

整体气质是现代、干净、轻技术感。中文大标题在深色背景上很稳，文章页阅读体验也不错。

### 留白与密度

首页和概念页留白充足，有“漂浮在深海中”的空间感。日志页和提示词页的信息密度更高，尤其移动端 `/log` 顶部的标签区会占用较长首屏空间，第一条内容出现较晚。

### 卡片、边框、阴影、背景

卡片主要使用深色半透明背景、细边框、弱阴影和金色 hover。没有过多装饰，整体是克制的。提示词卡片和文章页 PromptBlock 的边框、头部栏、复制按钮都比较成熟，适合长期沉淀可复用模块。

### 图形元素

最重要的图形元素有：

- 首页夜海视频 / 背景图。
- D3 概念图谱节点。
- 五层逻辑的圆形符号。
- 日志页时间段的小圆点和横线。
- Prompt 卡片的结构化容器。

整体风格判断：

- 克制中带丰富，不是极简白板。
- 技术感与人文感并存，偏人文叙事多一点。
- 既是叙事型，也是工具型的早期形态。
- 更接近“数字花园 / 个人知识空间”，不是传统博客。

## 5. 内容呈现方式

文章列表在 `/log` 采用混合卡片形式。不同内容类型有不同视觉处理：

- 随想像时间流里的短句，使用圆点和斜体。
- 文章使用深色卡片、日期、标签、摘要和“阅读 →”。
- 对话用 `W / AI` 小头像表达对话关系。
- 食谱用更轻的金色标识，和其他内容类型区分开。

这种设计优点是：日志列表不单调，能表达“这不是同一种内容”。缺点是：类型差异主要靠视觉，而不是结构说明。新访客可能不知道这些内容类型在知识空间中的角色。

单篇文章阅读体验整体舒适。`AI 辅助设计的工作流` 页顶部有日期和标签，正文宽度合适，PromptBlock 清楚地把可复用提示词从正文中分离出来。文章底部有“关联概念”，能把读者带回概念网络。

标签、分类、关联内容已经具备：

- 标签用于横向主题过滤。
- 概念用于语义归属。
- 概念详情页显示关联概念和关联内容。
- 提示词库从文章中的 PromptBlock 中抽取可复用模块。

目前内容网络仍有提升空间。它已经有“文章 → 概念”的连接，但还缺少更强的“内容之间互相引用”“概念为什么连接”“这条边来自哪些文章”的解释层。对于长期积累来说，底层模型是对的，下一步需要让关系从元数据变成可阅读、可操作、可被 agent 调用的知识对象。

## 6. 交互体验

当前已有的交互包括：

- 固定导航，滚动后加深背景和 blur。
- Astro 页面转场。
- 首页 Hero 随滚动淡出。
- D3 概念图谱：节点可点击、可拖拽、hover 显示标题和内容数。
- 日志页标签和概念筛选：使用 URL 参数和前端过滤。
- Pagefind 搜索弹窗：支持 `⌘K / Ctrl+K`。
- Prompt 卡片复制按钮。
- 移动端汉堡菜单。

这些交互都服务于“继续探索”。尤其概念图谱、日志筛选和提示词复制，是从博客向知识工具转型的关键。

明显负担主要有三类：

- 概念图谱缺少说明。用户能看到节点和连线，但不知道线代表什么、大小代表什么、金色节点和灰色节点的区别是什么。
- 日志筛选视觉权重偏弱。桌面和移动端的标签大量使用低透明度，未激活状态有点像不可用状态。
- 搜索在开发环境依赖构建后的 Pagefind 索引。弹窗文案已经考虑“搜索索引暂不可用”，但从产品体验看，搜索应该在无索引时提供替代入口，例如热门概念、最近内容或提示词库。

## 7. 移动端体验

移动端导航清楚，顶部保留 `Walker`、搜索图标和菜单按钮。菜单展开后显示 `港口 / 海域 / 概念 / 提示词`，文字足够大，点击区域也比较舒适。

移动端首页首屏视觉冲击力强，但它也带来两个问题：

- 首屏高度几乎完全被氛围图占据，知识入口全部在下方。
- 纸船主体被裁切，金色光路完整，但“船”的识别稍弱。

移动端日志页内容密度偏高。顶部标题、标签区、概念筛选区占用空间较多，第一条内容出现较晚。标签换行后的节奏还算整齐，但未激活标签对比度偏低。

移动端文章页是当前表现最好的页面之一。标题、日期、标签、正文行宽和 PromptBlock 都适合阅读。PromptBlock 在手机上没有溢出，复制按钮也保持可见。

移动端概念页结构清楚，但卡片摘要有截断，概念之间的“为什么相连”仍不明显。对于“概念图谱为核心”的目标，移动端需要更强的逐步浏览模式：不是把桌面图谱缩小，而是提供“当前概念 → 相邻概念 → 相关内容 → 可执行提示词”的纵向路径。

## 8. 可访问性与性能观察

字体对比度方面：

- 标题和正文对比度较好。
- 日期、摘要、说明文字、未激活标签对比度偏低，尤其在移动端和深海背景上更明显。
- 金色作为强调色效果好，但不宜承担太细小的文字信息。

点击区域方面：

- 导航和移动菜单点击区域基本足够。
- 日志筛选 pill 较小但仍可用。
- 图谱节点在移动端可能过小，不适合作为主要点击入口。

动画方面：

- Hero 视频、滚动淡出、hover、D3 力导向动画都不算过度。
- 但首页视频资源可能影响首屏加载体感，建议保留 poster、压缩版本和 `prefers-reduced-motion` 策略。

性能体感方面：

- 本地访问速度良好。
- 页面整体资源压力主要来自 Hero 视频、Google Fonts、D3 图谱和 Pagefind。
- 如果面向中文访问者，Google Fonts 可能带来不稳定加载，建议考虑本地化字体或系统字体优先。

SEO 和语义化方面：

- `lang="zh-CN"`、页面 title、description、JSON-LD、`data-pagefind-body` 都是正向信号。
- 概念和内容已经有结构化数据基础。
- 还可以补充 OG/Twitter metadata、`llms.txt`、概念图 JSON、RSS/Atom、站点地图中的概念关系说明，让人和 agent 都更容易消费。

## 9. 设计亮点

### 1. 夜海航行主视觉

具体是什么：首页用夜海、纸船、金色光路建立第一印象。  
为什么有效：它不是通用科技感背景，而是直接服务 Walker 的人格和“知识航行”隐喻。  
如何迁移：保留作为品牌入口，但增加首屏到知识图谱的低干扰引导，避免氛围压过任务。

### 2. `港口 / 海域 / 概念 / 提示词` 的导航隐喻

具体是什么：导航不是“首页 / 博客 / 分类”，而是统一在航行语义中。  
为什么有效：它让站点有完整世界观，不像模板博客。  
如何迁移：保留隐喻，同时用副标题、tooltip 或首次访问提示降低理解成本。

### 3. 概念作为核心数据结构

具体是什么：内容 frontmatter 中有 `concepts`，概念自身有 `related`、`domain`、`layer`。  
为什么有效：这是从博客升级为知识空间的根基。文章不再只是时间流，而是概念网络的证据。  
如何迁移：下一步把概念页做成“定义、来源、相关内容、相关提示词、反向链接、agent 摘要”的知识节点。

### 4. 五层逻辑作为阅读路径

具体是什么：首页和 `/framework` 把五个核心概念串成顺序。  
为什么有效：新访客需要入口，不只是自由探索。五层逻辑提供了“从哪里开始”的答案。  
如何迁移：把五层逻辑变成 onboarding 路径，例如“10 分钟理解 Walker 的知识系统”。

### 5. D3 概念图谱

具体是什么：首页有可点击、可拖拽的节点图，节点大小与内容数相关。  
为什么有效：它直接表达“内容之间有关系”，符合知识空间目标。  
如何迁移：增加图例、筛选、聚焦模式、边的解释，并允许从图谱进入具体内容证据。

### 6. 多类型日志卡片

具体是什么：文章、随想、对话、食谱等有不同卡片形态。  
为什么有效：内容类型差异被视觉表达出来，日志页更像生活和思考的混合航海图。  
如何迁移：继续保持类型差异，但给每类内容定义明确角色：证据、片段、对话、实践、工具。

### 7. PromptBlock 与提示词库

具体是什么：文章中的提示词块可以被抽取到 `/prompts`，并提供复制按钮。  
为什么有效：它把内容从“可读”推进到“可执行”。这是 AI 时代个人知识空间的重要方向。  
如何迁移：提示词不只展示文本，还应展示适用场景、输入、输出、验证标准和关联概念。

### 8. 文章底部关联概念

具体是什么：文章结束后展示关联概念 pill。  
为什么有效：读者读完一篇文章后能回到概念网络，不会停在单篇内容。  
如何迁移：增加“为什么关联”“同概念下的下一篇”“该概念的核心问题”，让跳转更有上下文。

### 9. 命令式搜索体验

具体是什么：桌面端有 `⌘K / Ctrl+K` 搜索弹窗。  
为什么有效：对知识空间而言，搜索是回访用户和 agent 风格用户的重要入口。  
如何迁移：搜索弹窗可以升级为 command palette，支持“搜索内容、跳转概念、复制提示词、打开关系图”。

### 10. 克制的深色卡片系统

具体是什么：全站大量使用深色半透明卡片、细边框、弱金色 hover。  
为什么有效：保持夜海氛围，同时不会变成花哨界面。  
如何迁移：保留深色卡片，但为不同 domain 加少量辅助色，避免长期阅读中只有深蓝和金色两个层次。

## 10. 设计问题

### 1. 隐喻强，但新访客理解成本偏高

问题在哪里：`港口`、`海域` 等命名有风格，但不够直接。  
为什么影响体验：第一次访问者可能不知道“海域”就是日志，“港口”就是首页。  
改进建议：保留主命名，增加次级说明，例如 `海域 / 日志`、`概念 / 图谱`，或在首页加入一句“从概念图谱、航海日志和提示词库进入我的知识系统”。

### 2. 首页首屏过于完整地让位给氛围

问题在哪里：首屏几乎只有品牌视觉，深层内容入口在滚动后。  
为什么影响体验：视觉记忆强，但任务路径弱。  
改进建议：在不破坏画面的前提下，增加底部滚动提示或三个轻量入口：概念图谱、最近痕迹、提示词库。

### 3. 图谱当前更像视觉符号，还不是知识工具

问题在哪里：节点、大小、颜色、连线含义没有解释。  
为什么影响体验：用户知道“有关系”，但不知道“是什么关系”。  
改进建议：加入图例、节点详情面板、边的来源、按 domain/layer 过滤、搜索定位节点。

### 4. 概念详情页信息深度不足

问题在哪里：`/concept/subtraction` 有定义、关联概念和关联内容，但概念本身还偏短。  
为什么影响体验：如果概念是核心，详情页应该是知识节点，而不只是分类页。  
改进建议：每个概念页增加固定结构：一句定义、核心问题、反例、实践场景、相关提示词、关联内容、更新历史、agent 摘要。

### 5. 日志筛选区在移动端占屏过多

问题在哪里：移动端 `/log` 顶部标签和概念筛选占据很大空间，第一条内容出现较晚。  
为什么影响体验：回访用户要先越过筛选区才能读内容。  
改进建议：移动端改为折叠筛选、横向滚动 chips，或默认只显示热门标签，展开后再显示全部。

### 6. 辅助文字对比度偏低

问题在哪里：说明文字、摘要、日期和未激活标签大量使用低透明度。  
为什么影响体验：深色背景下容易显得“不可读”或“不重要”。  
改进建议：提高正文辅助信息到至少 `parchment/50` 左右，未激活筛选保持弱化但不要像 disabled。

### 7. 搜索体验依赖构建索引

问题在哪里：Pagefind 在未构建时会提示索引不可用。  
为什么影响体验：本地开发或预览阶段搜索不可用，用户打开后容易失望。  
改进建议：无索引时展示最近内容、热门概念、提示词入口；生产环境确保构建流程稳定生成索引。

### 8. 文章页缺少长文辅助阅读组件

问题在哪里：当前文章页有正文和关联概念，但没有目录、阅读进度、上下篇或同概念推荐。  
为什么影响体验：短文没问题，长期积累后深度文章会变难导航。  
改进建议：给长文增加浮动目录、阅读进度、同概念下一篇、返回概念节点。

### 9. 内容之间的网络还不够显性

问题在哪里：现在主要是“内容绑定概念”，但“内容引用内容”“概念边来自哪些内容”还没有显现。  
为什么影响体验：知识网络的价值没有完全释放。  
改进建议：增加反向链接、引用关系、同船记录、关系证据列表，让每条连接都可追溯。

### 10. AI agent 友好性还停留在早期

问题在哪里：已有 JSON-LD 和内容 schema，但没有专门给 agent 的索引、图谱 JSON、摘要或操作入口。  
为什么影响体验：人类能读，agent 还需要自己解析页面。  
改进建议：增加 `/llms.txt`、`/knowledge-graph.json`、概念级 machine summary、提示词结构化字段、内容关系 API。

## 11. 对 Walker 知识空间下一步的启发

Walker 当前已经具备三个很重要的基础：

- 视觉上有独立人格：夜海、行船、金色光路。
- 数据上有概念模型：概念、关系、内容类型、标签。
- 交互上有知识工具雏形：图谱、筛选、搜索、提示词复制。

下一步的核心不是“再像博客一点”，而是让它从“有知识空间气质”升级为“真的可以操作知识的空间”。

值得借鉴并强化的设计原则：

- 每个页面都应该回答“我现在在哪片海域、和哪些概念相连、下一站可以去哪”。
- 概念不是分类，而是知识节点；文章不是列表项，而是概念的证据和案例。
- 提示词不是附属内容，而是可执行的知识模块。
- 图谱不是装饰，而是导航、解释和 agent 数据源。
- 夜海视觉应该保留，但不能让氛围阻挡进入内容。

不应该继续放大的地方：

- 不要让隐喻压过信息清晰度。
- 不要让首页长时间停留在纯品牌展示。
- 不要把图谱只做成“看起来像网络”的视觉组件。
- 不要让所有信息都长期停留在深蓝 + 金色的单一层级。

可以迁移或强化的结构：

- 首页：保留 Hero，但加入“概念图谱 / 最近痕迹 / 提示词库”的首屏入口。
- 概念页：升级为知识节点模板，而不是概念列表。
- 日志页：继续保留时间流，但加强概念筛选和类型解释。
- 提示词库：作为 AI 时代知识空间的核心资产，增加场景、输入、输出、验证字段。
- Agent 层：增加机器可读索引，让 AI 能直接读取概念、内容、关系和提示词。

可以参考的交互：

- 图谱聚焦：点击节点后显示定义、关联内容、相关提示词和相邻节点。
- Command palette：搜索之外，支持直接跳转概念、复制提示词、筛选日志。
- 关系证据：点击概念边，显示哪些文章让这两个概念发生连接。
- 阅读续航：文章底部不只返回日志，而是给出“继续沿这个概念阅读”。

视觉语言可以这样继续转译：

- 深海蓝继续作为底色，代表未知、深度和长期沉淀。
- 金色只用于方向、当前状态和关键判断，不要滥用。
- 概念 domain 可以引入非常克制的辅助色，例如技术偏冷青、厨艺偏暖白、设计偏淡紫灰，但饱和度要低。
- 图谱线条可以更像航线或星图，而不是通用 network graph。
- 移动端减少大面积留白和过长首屏，让“船”和“下一站”同时出现。

最关键的问题是：Walker 要从传统博客升级为 AI 时代个人知识空间，就要让“内容”变成“可寻址、可组合、可执行、可解释”的对象。当前站点已经有对象模型，下一步要把这些对象的关系和用途明确呈现出来。

| 移动端体验 | 7.5 | 导航和文章阅读良好；首页首屏过长、日志筛选占屏、部分低对比文字影响效率。 |

> 这个网站最值得保留的是：夜海航行的品牌叙事，以及“日志 + 概念 + 提示词”的三层知识模型。  
> 最不应该继续放大的是：把概念图谱停留在氛围化视觉入口，而没有变成可解释、可操作、可被 AI agent 消费的知识系统。
