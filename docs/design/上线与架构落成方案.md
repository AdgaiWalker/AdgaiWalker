# Walker 上线与架构落成方案(递进式)

> **日期**:2026-06-22
> **状态**:候选设计输入 — 待 Walker 批准后进入执行三件套
> **定位**:本文是 Walker 个人网站从"现在"到"NorthStar 拆站"的**单一权威路线文档**。它吸收 `落成.md`(API 治理目标态)和 `后端架构与技术栈演进方案.md`(技术栈演进候选),把它们切到正确的时间轴上。不另起 PRD,不分散到多个文件。
> **北极星**:个人网站先以最小护栏上线,治理在流量小的时候低成本推进,平台化时再借势完成最后一步。
> **使用方式**:按 P0 → P5 顺序读。每个阶段都有"目标 / 范围 / 任务 / 验收 / 不做"四件套,任务带修改面与自主决策边界。

---

## 0. 定位与边界

```text
个人网站阶段(现在)        →  验证自用闭环、内容宇宙、需求匹配
平台期(未来)              →  NorthStar 多人网络,迁到独立平台
个人网站长期形态           →  Walker 自己的作品集 + 自留地,长期存在
```

这个定位决定三件事:

1. **个人网站要能长期维护,不能堆技术债** —— 治理要做,但不阻塞上线。
2. **核心数据契约要从现在就稳** —— 否则迁移平台时会咬人。
3. **NorthStar 复杂度不能倒灌** —— 个人站永远不承担社区复杂度。

一句话:**先让网站上线跑起来,边跑边治理,比一次性治理完再上线风险低得多。**

---

## 1. 架构原则(全阶段贯穿,不可动摇)

| 原则 | 在本项目中的具体形态 |
|---|---|
| **按业务能力切分** | 六域:`workbench / needs / creation / assets / system / northstar`。不按 Controller/Service/DAO 分 |
| **清晰协议** | 所有跨边界调用走 `src/stores/ports.ts` 的 Port 接口。模块间不直接 import 对方 store |
| **隔离变化** | 核心域模型(NeedCase/WorkItem/Asset)稳定;多变的是 AI Gateway、存储实现、媒体策略 |
| **依赖倒置** | 上层 service 只依赖 Port,不依赖 Redis/Blob 具体实现 |
| **统一语言** | 代码术语与业务术语一一对应:`placeOrder` 不叫 `createTransaction` |
| **可逆决策** | PaymentProvider/AI Gateway/MediaStorage 都用 Port + 合成适配器,真实凭据接入时不动核心 |
| **识别偶然复杂度** | `verify-production-storage.mjs` 跨进程读回检查是必要的;三层线程池那种是该消除的 |

## 2. 六阶段路线总览

```text
P0 上线阻断            (现在 → 上线,1-2 周)
  凭据 + 信任链 + cron + 烟测;不动 API 路径

P1 API 治理底座        (上线后 1-2 天)
  api-registry + 测试移出 + system-map 读 registry

P2 workbench/needs 归位 (上线后 2-4 周)
  两域路径迁移 + schema 版本化

P3 平台期延伸域        (接 NorthStar 时,数月后)
  creation/assets/system/northstar 路径迁移

P4 public + v1         (NorthStar 上线时)
  公开 API 迁移 + 引入 /api/v1

P5 NorthStar 拆站      (NorthStar 成熟时)
  个人站保留为作品集 + 自留地
  通过 Publish Interface 选择性发布到独立平台
```

每个 P 都是独立可验收的产物,**不按 P 顺序推进就会累积技术债**。但 P1-P2 之间可以重叠,P3-P5 必须按顺序。

## P0. 上线阻断(现在 → 上线)

**核心:护栏 + 真实性,不动任何 API 路径。** 按重要性排。

### P0-1 真实外部凭据(Human Gate,Walker 操作)

```text
UPSTASH_REDIS_REST_URL / TOKEN   点赞/NeedCase/WorkItem/画像/邀请码持久化
BLOB_READ_WRITE_TOKEN            /admin/appearance 媒体上传
COOKIE_SECRET                    登录/注册签名(不配则直接抛错)
```

**风险点**:`storage-mode` 的降级路径会静默写内存。代码门闩已建(S0-03/S0-04),但要靠 verify 脚本兜底。

### P0-2 生产等价验证(真凭据下必须绿一次)

```bash
npm run check:production-readiness
npm run verify:production-storage       # 跨进程 Redis 读回 + 索引完整性
npm run verify:production-media-storage # Blob 写/读/删/跨进程
```

**没在真凭据下绿过一次 = 没护栏。** 这是 Vercel serverless 的命门:每个请求可能是新进程,必须证明 A 函数写的东西 B 函数读得回来。

> **当前状态(2026-06-22)**:第十三轮已用本地 ioredis + 文件系统完成"本地等价验证",解除 S0-03/S0-04/UX1-07 三项阻断。真实 Upstash / Blob token 下的云端加固仍未完成 —— 本地等价证明代码层契约正确,不等于 CDN 多区域复制 / 私有 URL / 真实 multipart 的云端行为。

个人站与 NorthStar 共享的是数据契约:

```text
Evidence → Decision → Action → Outcome → Capability
```

不共享代码库。各自演进,通过 Publish Interface 同步。

### P5-3 个人站长期形态

```text
个人站 = Walker 的作品集 + 自留地 + 个人 Agent 实验场
  - 公开内容、思考、工具、教程、生活切片
  - 邀请制轻会话 + 需求匹配
  - 内容宇宙多视图
  - 不承担社区复杂度
```

### P5-4 Publish Interface

通过 Publish Interface 从个人站选择性发布:

```text
- capability  (我会什么,已验证)
- offer       (我提供什么,可商业化)
- request     (对方需要什么,可匹配)
- engagement  (协作进行中)
- outcome     (结果回流,验证 capability)
```

### P5 验收标准

```text
1. NorthStar 独立平台运行
2. 个人站与 NorthStar 通过 Publish Interface 同步
3. 个人站不承担社区复杂度(账号/信息流/推荐/审核)
4. 两端数据契约一致,代码库独立
5. 个人站主路径仍完整运行
```

## 7. 最终判断

> **这份方案的本质是把 `落成.md` 的"正确目标态"切成六段,让个人网站先以最小护栏上线,治理在流量小的时候低成本推进,平台化时再借势完成最后一步。** 现在的关键不是"做更多",而是"做对的那几件"。

```text
P0 上线前:  凭据 + 信任链 + cron + 烟测        (1-2 周)
P1-P2 上线后: 治理底座 + workbench/needs 归位  (2-4 周)
P3-P5 平台期: 剩余四域 + public + v1 + 拆站   (数月后)
```
