# Walker API 治理落成方案

Status: Proposed / 待实施  
Updated: 2026-06-22  
Scope: AdgaiWalker 后台与前台 API 治理  
Owner: Walker

## 1. 结论

当前后台模块和主要后端能力已经写完，但 API 路由仍保留了“页面需要什么就长一个接口”的痕迹。下一步不是继续补页面，也不是一次性重写全部后端，而是把 API 升级为可治理的系统契约。

目标从：

~~~text
页面需要接口 -> 新增一个路由
~~~

改成：

~~~text
谁调用
属于哪个责任域
读写什么事实
是否改变权威状态
是否需要审计
完成后进入哪个下一步
~~~

本方案采用“先登记、再迁移、后删除”的路线，不写兼容代码；每轮迁移一个责任域，确保页面调用、测试、文档和审计一起收口。

## 2. API 总体结构

顶层分为三个信任边界：

~~~text
/api/public/*      访客、前台页面、公开互动
/api/admin/*       Walker 后台、Owner、未来 Contributor
/api/internal/*    定时任务、批处理、AI job、系统内部作业
~~~

后台再按责任域拆分：

~~~text
/api/admin/workbench/*
/api/admin/needs/*
/api/admin/creation/*
/api/admin/assets/*
/api/admin/system/*
/api/admin/northstar/*
~~~

说明：

- workbench 只拥有决定、行动和结果，不拥有第二份原始事实。
- needs 拥有用户、需求、反馈、Walker 主张和选题。
- creation 拥有内容、简报、发布和命中率。
- assets 拥有经验、规则、Skill、学习任务和证据链。
- system 拥有登录、权限、AI Gateway、事件、审计、外观和媒体。
- northstar 是范围扩展，不塞进 system；默认关闭，关闭时所有写操作拒绝。

暂不启用 /api/v1。当前系统还不是对外开放平台，全量版本号会提前增加迁移成本。等出现稳定外部消费者、Webhook 或第三方接入时，再引入 /api/v1。

## 3. API Registry

第一步新增：

~~~text
src/lib/api-registry.ts
src/lib/api-registry.test.ts
~~~

每个现有路由必须登记：

~~~ts
{
  currentPath: "/api/admin/accounts/delete",
  targetPath: "/api/admin/needs/accounts/:username/delete",
  surface: "admin",
  domain: "needs",
  kind: "command",
  ownsFact: true,
  audit: "high-risk",
  status: "replace"
}
~~~

建议类型：

~~~ts
type ApiSurface = "public" | "admin" | "internal";

type ApiDomain =
  | "workbench"
  | "needs"
  | "creation"
  | "assets"
  | "system"
  | "northstar";

type ApiKind =
  | "intake"
  | "query"
  | "projection"
  | "command"
  | "job";

type AuditLevel =
  | "none"
  | "write"
  | "high-risk";

type ApiStatus =
  | "keep"
  | "rename"
  | "replace"
  | "split"
  | "retire";
~~~

Registry 测试必须保证：

~~~text
1. src/pages/api 下每个实际路由都在 registry 登记
2. admin API 必须声明 domain
3. command 必须声明 audit 等级
4. high-risk command 必须有 preview/confirm 或明确审计理由
5. 测试文件不得出现在 src/pages/api
6. public API 不允许修改后台权威对象
7. northstar 写接口必须受 disabled guard 保护
~~~

## 5. AI API 的位置

不建立独立的 /api/admin/ai/*。

AI 不是业务域，而是 proposal 能力，挂在各责任域中：

~~~text
POST /api/admin/workbench/decisions/proposals
POST /api/admin/needs/topics/proposals
POST /api/admin/assets/skills/proposals
POST /api/admin/creation/briefs/proposals
~~~

统一字段：

~~~ts
{
  proposedBy: "ai",
  model: string,
  evidenceRefs: EvidenceRef[],
  uncertainty: string,
  expiresAt: string,
  requiresHumanDecision: true
}
~~~

AI 可以生成建议，不能绕过人工决定。

## 6. 统一响应格式

新增：

~~~text
src/lib/api-response.ts
~~~

成功：

~~~ts
{
  ok: true,
  data,
  meta: {
    requestId,
    generatedAt
  }
}
~~~

失败：

~~~ts
{
  ok: false,
  error: {
    code,
    message,
    details
  },
  meta: {
    requestId
  }
}
~~~

写接口额外返回：

~~~ts
{
  auditId,
  affectedObjects,
  nextActions
}
~~~

## 7. 高风险动作

以下动作必须进入高风险治理：

~~~text
删除内容
删除用户
改权限
发布到 NorthStar
Skill admission
真实支付 / 退款
隐私数据导出
~~~

统一形态：

~~~text
POST /xxx/:id/preview
POST /xxx/:id/confirm
~~~

或：

~~~text
POST /xxx/:id/delete-preview
POST /xxx/:id/delete-confirm
~~~

返回必须包含：

~~~ts
{
  impact: string[],
  rollback: string,
  auditRequired: true,
  confirmRequired: true
}
~~~

## 9. 实施顺序

~~~text
P0：api-registry + 测试文件移出路由目录
P1：workbench 路由归位
P2：needs 路由归位
P3：creation 路由归位
P4：assets 路由归位
P5：system / northstar 路由归位
P6：public API 归位
P7：统一响应格式与高风险 preview/confirm
~~~

每一阶段都必须：

~~~text
改页面调用
改 API 路由
删除旧入口
更新 api-registry
更新测试
跑完整验证
~~~

不保留旧路由兼容别名，除非有明确外部消费者证据。

## 10. 验收标准

完成后必须满足：

~~~text
1. src/pages/api 下没有 *.test.ts
2. 每个 API 都在 api-registry 登记
3. 每个 admin API 都有 domain
4. 每个 command 都有 audit 等级
5. 每个 high-risk command 都有 preview/confirm 或明确例外
6. public API 不读取或修改后台权威对象
7. workbench 不拥有第二份原始事实
8. NorthStar disabled 时所有 northstar 写操作拒绝
9. 页面端不再调用旧路由
10. 全量测试、构建、E2E 通过
~~~

## 11. 第一张 PR 范围

第一张 PR 只做治理底座，不做大规模业务迁移：

~~~text
新增 src/lib/api-registry.ts
新增 src/lib/api-registry.test.ts
登记所有当前 API
移走 src/pages/api 下的测试路由
让 admin-system-map 从 registry 读取 API 归属
新增本文件作为实施依据
~~~

第一张 PR 完成后，后续每个模块迁移都必须先过 registry 测试，避免 API 再次无序生长。

## 12. 不做

本轮 API 治理不做：

~~~text
不一次性全量迁移 60+ 路由
不引入 /api/v1
不新增外部开放平台能力
不为旧路由写兼容 alias
不让 AI 自动执行高风险动作
不把 NorthStar 混入 Walker 个人后台核心 API
~~~

## 13. 最终判断

API 治理的落成标准不是“路径更整齐”，而是：

~~~text
每个接口都能回答：
谁能调用？
属于哪个责任域？
读写什么事实？
是否改变权威状态？
是否需要审计？
执行后进入哪个下一步？
~~~

答不上来的接口，要么登记为待证明，要么迁移，要么删除。
