# 设计命题书：Walker 后台重设计

> 这不是设计规范，是一份创作挑战。
> 给命题，不给解法。用你的风格，给我惊喜。

---

## Walker 是什么

Walker 是一个 **AI 驱动的个人 Agent 系统网站**（iwalk.pro）。

作者是一个自媒体创作者。他把自己沉淀的知识、工具、经验、生活现场，存进这个系统——既给自己用（像一个会生长的数字分身），也对外开放（访客来消费内容、提问、被 Agent 匹配答案）。

访客从自媒体来，点进网址。有的人看文章，有的人用提问功能找答案，少数人会认证成用户。流量不大，但每一份都是真实的。

这个项目的隐喻是 **Ferry（渡船）/ 航海日志**。作者把人与 AI、点子与行动、当下经验与长期系统的关系，理解成一艘船——不是替你抵达彼岸，而是把你带到更适合行动的位置。

## 前台已经存在，先去看

公开站是活的。去看它，感受它的语言：

- 主色翡翠青 `#35bfab`，是这套系统的 DNA
- 有玻璃感、流动光球、雾气叠层——是"航海"的浪漫
- 命名是海的意象：abyss（深渊）、sea、parchment（羊皮纸）、stardust（星尘）
- 四套主题：流光霜雾 / 极光霓虹 / 落日余晖 / 数码薄荷

**看完，用你自己的方式演化。别照搬。** 我要的是同一艘船的不同舱室，不是另一个产品。

## 后台现在的问题

后台是只给作者和未来的 contributor 看的——**驾驶舱，不是展厅**。

但现在的后台是另一个世界：固定赤陶色（跟前台的翡翠青毫无关系），17 个页面平铺在一条横栏里各自为政，每个页面自己手写一套样式，没有共享骨架。访客进来看到的是一艘会发光的海船，管理员一转身进后台，像进了一间赤陶色的会计室。

它的心智模型也是错的——它被当成"内容管理器"在做（CRUD 文章、管用户），但作者真正需要的，是一个让**「访客需求 → 内容产出 → 验证效果 → 再调整」**这条闭环跑起来的工作台。每一份后台的数据、每一条访客反馈，最终都要能变成一个动作：**改内容，或改功能**。

## 这个系统有哪些部分

后台不是一个抽屉柜，是一台机器——几个部件咬合在一起转。下面是它的功能分区图：每个方块负责什么、它们之间怎么连。**这是结构，不是页面设计——方块长成什么样、怎么摆，是你的事。**

```
① 工作台（首页）
   "现在该干啥"的总览。把下面所有模块的待处理信号，汇成"今天该做的几件事"。

② 创作循环（对外的那条线）
   访客问什么 → 我答了没 → 答得好不好 → 还该写什么
   ├ 洞察      采集访客信号（需求大盘 / 功能使用 / 文章热度 / 反馈）
   ├ 选题库    把信号聚成"该写什么"
   ├ 内容管理  产出（写 / 改 / 藏）
   └ 命中率    验证产出有没有真解决问题（反馈回看）

③ 资产沉淀（对内的那条线）
   经验 → 规则 → Skill，沉淀成 Agent 可调用的资产
   ├ 经验     原始事件采集
   ├ 规则池   模式提取（observed → stable）
   └ Skill    准入注册（反复有效 / 边界清晰 / 输出可验证）

④ 基建（让前两条线跑起来）
   ├ AI Gateway  服务健康台（成功率 / 延迟 / 成本 / 限流 / 降级）
   ├ 访问控制    用户 + 邀请码（邀请制身份入口）
   └ 事件        安全 / 降级待复盘

⑤ Agent 治理（未来席位，现在留位）
   Agent 调用了什么 / 调得对不对 / 该补什么给它
```

### 部件之间怎么咬合

```
访客（自媒体引流）
    │ 行为 + 反馈
    ▼
┌─ 洞察（采集信号）──────────────────────────┐
│  功能使用 / 文章热度 / 反馈 / 需求大盘       │
└──────────────────┬─────────────────────────┘
                   │ 信号归一
                   ▼
            选题库（决策：写什么 / 改什么 / 删什么）
                   │
         ┌─────────┼──────────┐
         ▼                    ▼
      内容管理           （删功能 → 影响前台）
         │ 产出
         ▼
       发布上线 ──→ 命中率（验证效果）
         │              │
         │              │ 反馈回流
         │              ▼
         │           回到「洞察」（下一轮信号）
         │
         │          ┌──────────────────┐
         └──────────┤   AI Gateway     │← 支撑所有 AI 调用
                    │   （命脉）        │
                    └──────────────────┘

   资产沉淀 ──────→ ┌──────────────────┐
   经验→规则→Skill │   Agent 治理      │（未来）
                    │   调用这些资产    │
                    └──────────────────┘
```

这张图想传达三件事：

1. **后台是一个循环**，不是 17 个并列抽屉。访客的信号进来，沿着"洞察 → 选题 → 内容 → 命中率"转一圈，再回到洞察——每转一圈，系统和内容都长一点。
2. **洞察是枢纽。** 所有信号先进它，再分流到别处。它是后台的入口和心脏。
3. **每个模块都通向一个动作。** 改内容、改功能、沉淀资产——没有哪个模块是"纯展示、看完就走"的。一个数据如果带不出一个动作，它就不该出现在后台。

## 我要什么

以需求为导向。实用第一，好看第二。

后台要让人一进来就知道**该做什么**——今天哪些需求在排队、哪篇内容该改、哪个功能没人用该删、AI Gateway 是不是在降级。不是看一堆数字满足好奇心，是看数据为了动手改东西。

我要它**只属于这个项目**。它可以是深色的、可以是亮色的、可以是我想都没想过的形态——但它必须跟前台是同一个生命体，不是贴上去的补丁。

## 给你

**用你的风格，给我一个惊喜。**

不要参考 SaaS 模板。不要参考通用后台。不要做成"看起来很专业但谁都能用"的样子——那种东西网上有一万个，我不需要再多一个。

我想看到的是：一个只服务于"AI 驱动的个人创作系统"这个具体命题的后台，它的形态是我没想过会这么做、但看到就觉得"对，就该是这样"的。

先给我 2-3 个页面，让我看到方向。方向对了，我们再往下走。
