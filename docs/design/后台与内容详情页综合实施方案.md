# Walker 后台与内容详情页综合实施方案

Status: Planning / Candidate  
Updated: 2026-06-20  
Scope: AdgaiWalker 个人系统  
Owner: Walker

> 本文统一后台决策系统与内容详情页的模块责任、数据关系、实施顺序和验收标准。它是候选实施方案，不代表其中功能已经实现。选定的阶段进入 `walker-northstar/references/working/` 工作区后，才拥有当前执行权。

## 1. 核心结论

后台与内容详情页不是两个并列项目。

- 后台负责把需求、主张、反馈和系统事件变成决定与行动。
- 内容详情页负责交付内容，并采集访客是否真的解决了问题。
- 结果数据回到后台，用于决定修改内容、补教程、改功能、停止维护或沉淀资产。

统一产品契约：

```text
Evidence
-> Decision
-> Action
-> Published Result
-> Outcome
-> Capability
-> New Evidence
```

## 2. 优先级评判规则

不按页面数量、视觉完成度或开发方便程度排序。统一按以下顺序判断：

```text
1. 安全、权限和服务可用性
2. 数据是否真实、持久、可追溯
3. 是否打通每日高频闭环
4. 是否能产生可验证结果
5. 是否解除后续模块的共享依赖
6. 是否减少重复工作并沉淀能力
7. 最后才是观感增强、协作、公共网络和交易
```

不同性质的事项先分队列，再在同类队列内排序：

| 队列 | 判断依据 |
| --- | --- |
| 用户需求 | 影响范围、频率、原话证据、时效、未解决程度、行动成本 |
| Walker 主张 | 长期方向、个人承诺、学习价值、当前时机、可验证性 |
| 系统事件 | 安全、隐私、可用性、波及范围、紧急度、可逆性 |
| AI 资产 | 复用频率、结果稳定性、边界缺口、错误风险、补证成本 |

## 3. 模块责任

### 3.1 一级模块

| 模块 | 拥有的责任 | 拥有的事实 | 不应负责 |
| --- | --- | --- | --- |
| 工作台 | 汇总待决定、待执行、待验证和已阻塞事项 | 不拥有原始事实；只保存决定、行动及其状态 | 复制用户、内容、反馈或系统数据 |
| 需求 | 管理用户、需求原话、反馈、Walker 主张、需求簇和选题 | `NeedCase`、`Feedback`、`WalkerThesis`、`TopicCandidate` | 编辑或发布内容 |
| 创作 | 管理简报、文章、功能改动、发布状态与内容命中 | `Content`、`Brief`、发布记录及内容关联 | 定义需求的原始身份 |
| 资产 | 管理知识来源、真实经验、规则、Skill、学习任务和 Agent 调用 | `KnowledgeSource`、`Experience`、`Rule`、`Skill`、`LearningRequest` | 把未验证材料直接注册为 Skill |
| 系统 | 管理登录、权限、Gateway、遥测、事件、日志、主题和媒体 | 运行状态、授权、审计、配置和媒体元数据 | 决定创作方向 |

### 3.2 内容详情页的责任

内容详情页是创作模块的公开交付面，同时是 Outcome 的采集入口，不是新的后台一级模块。

| 部件 | 责任 | 不负责 |
| --- | --- | --- |
| `ContentShell` | 阅读容器、TOC、进度、反馈区、评论区 | 探测首块类型或替内容块决定宽度 |
| 内容 Block | 表达视频、提示词、对话、步骤、资源等局部语义 | 保存后台决定 |
| `BlockFeedback` | 提交“有用 / 需补充 / 已过时”等明确信号 | 自行计算命中率 |
| `BlockLike` | 采集轻量喜欢信号 | 代替“问题是否解决”的判断 |
| Feedback API | 验证、脱敏、持久化反馈，建立内容与来源关联 | 在浏览器本地冒充持久化 |
| hit-rate | 从真实反馈和来源关联派生结果 | 拥有第二份反馈事实 |

### 3.3 跨模块共享对象

| 对象 | 用途 |
| --- | --- |
| `EvidenceRef` | 指向原始需求、反馈、事件、内容或调用结果，不复制原文 |
| `DecisionItem` | 记录当时要决定什么、依据什么以及 Walker 的选择 |
| `ActionItem` | 记录由谁执行什么、预期结果、验证时间和回滚方式 |
| `OutcomeEvent` | 记录现实结果、证据和对原决定的修正 |
| `LearningRequest` | 证据不足时，要求 Walker 补充实践、反例、访谈或结果 |

## 4. 模块关系

### 4.1 需求到内容结果

```text
访客访问 / 搜索 / 提问 / 反馈
        -> EvidenceRef
        -> NeedCase / Feedback
        -> TopicCandidate
        -> DecisionItem
        -> ActionItem
        -> Brief / Content / Feature Change
        -> 发布到内容详情页
        -> 明确反馈 + 必要遥测
        -> OutcomeEvent
        -> 修改 / 维持 / 停止 / 补证 / 资产候选
```

### 4.2 Walker 主张与用户需求

```text
WalkerThesis --------------------\
                                  -> converged TopicCandidate
NeedCase / Feedback -------------/
```

- 两边可以汇合，但原始对象不删除。
- `demandSourceIds` 与 `walkerThesisId` 分别保留。
- Walker 主张不得写成“用户高频需求”。
- 后续 Outcome 可同时回写两边，但不改变其来源身份。

### 4.3 结果到资产

```text
OutcomeEvent
  -> 一次有效：Experience / case
  -> 重复有效：Rule candidate
  -> 边界清晰、输出可验证：Skill candidate
  -> Walker 批准后：registered Skill
```

AI 可以生成候选、评测和缺口分析，不能自行将未验证材料升级为 Walker 能力。

## 5. 真实数据契约

每条进入后台的事实或派生判断至少带有：

```text
sourceType
sourceId
occurredAt
collectedAt
environment
visibility
consent / lawfulBasis
freshness
qualityStatus
```

真实性规则：

1. 数据缺失显示“未采集 / 无法判断”，不用静态数字补位。
2. AI 推断使用 `proposal` 或 `hypothesis` 身份，不冒充事实。
3. 生产数据必须持久化；进程内存只能用于明确标记的开发模式。
4. `localStorage` 可记录 TOC 折叠、主题偏好等客户端偏好，不保存决定、行动、反馈或 Outcome 的权威状态。
5. 工作台、hit-rate 和数据分析只做投影，不建立新的原始事实库。
6. 每项数据必须能显示来源、时间、统计范围和更新状态。

## 6. 综合实施顺序

### H0：已知问题快速收口

目标：清除内容详情页已经确认的功能错误和单一真相源问题。

| ID | 任务 | 依赖 | 验收 |
| --- | --- | --- | --- |
| H0-01 | 把 TOC 自动滚动容器从 `.ghost-toc-inner` 修正为 `.toc-sidebar-inner` | 无 | 活跃目录项会自动进入可视区 |
| H0-02 | 删除 `.cs-layout` 和 `.toc-fab` 死 CSS | 无 | 全局零引用，视觉无变化 |
| H0-03 | 用 `--cs-toc-offset` 取代两处 `362px` | H0-02 | TOC 位置等价 |
| H0-04 | 抽取 `setCollapsed()` | 无 | 折叠、展开和偏好记忆正常 |
| H0-05 | 删除未使用的 `BLOCK_WIDTH_STYLES` | 无 | CSS 变量成为宽度唯一真相源 |
| H0-06 | 统一 `BlockDialogue` class 写法并修正过时注释 | H0-03 | 构建通过，视觉无变化 |

该批次可以先完成，但不代替 P0 产品基础。

### P0：真实、安全、可持久

目标：任何页面都不使用假状态影响 Walker 决策。

| ID | 任务 | 依赖 | 验收 |
| --- | --- | --- | --- |
| P0-01 | 盘点每个后台字段的事实源，删除硬编码状态和无来源指标 | 无 | 空数据显示空状态，不显示假数字 |
| P0-02 | 建立 `EvidenceRef`、`DecisionItem`、`ActionItem`、`OutcomeEvent` 最小持久化存储 | P0-01 | 刷新、重启和跨页后状态不丢失 |
| P0-03 | 把工作台中的决定、行动和结果从 `localStorage` 迁移到持久化契约 | P0-02 | 浏览器本地不再拥有业务权威状态 |
| P0-04 | 完成 Owner 登录、访问控制与审计基线 | 无 | 未授权不可读写管理数据，开发预览不影响生产 |
| P0-05 | Gateway 健康、降级和系统事件只读取真实运行数据 | P0-01 | 无数据时显示未知，不硬编码“系统可用” |
| P0-06 | 明确开发内存 fallback 与生产持久化的环境标识 | P0-02 | 生产禁止静默降级为进程内存 |
| P0-07 | 高风险动作加入预览、确认、审计和回滚说明 | P0-02, P0-04 | 发布、删除、权限和隐私动作不能静默执行 |

P0 验收锚点：Walker 能安全进入后台；每个数字和状态都能说明来源；决定、行动和结果不会因刷新或重启消失。

### P1：需求到内容结果的真实闭环

目标：让一条真实需求或 Walker 主张走完一个可验收周期。

| ID | 任务 | 依赖 | 验收 |
| --- | --- | --- | --- |
| P1-01 | 定义并实现单条内容反馈写入 API | P0-02, P0-04 | 反馈经验证后持久化，重启后仍可查 |
| P1-02 | 实现 `BlockFeedback`，优先采集“有用 / 需补充 / 已过时” | P1-01 | 提交后后台可见原始反馈及时间 |
| P1-03 | 把 `BlockFeedback` 接入 `ContentShell` 固定反馈区 | P1-02 | 所有目标内容的反馈入口一致 |
| P1-04 | 打通 `NeedCase -> TopicCandidate -> Content -> Feedback` 关联 | P1-01 | 任一反馈可回溯内容与原需求；未关联时显示未知 |
| P1-05 | hit-rate 从真实关联派生 Outcome，不把缺失数据计为失败 | P1-04 | 能解释分子、分母、时间范围和未知数 |
| P1-06 | 工作台按真实 Evidence 生成待决定项 | P0-02, P1-04 | 任一事项可打开原始证据和优先理由 |
| P1-07 | 决定形成 Action，完成后记录 Outcome 并回写原对象 | P1-05, P1-06 | 一条需求走完 Evidence -> Decision -> Action -> Outcome |
| P1-08 | 接入 `BlockLike` 作为辅助信号 | P1-03 | 点赞不代替明确反馈或 Outcome |

P1 验收锚点：后台不只能看到内容被访问，还能判断它是否解决了来源问题，并立即创建下一步动作。

| P2-04 | 清理 `videos/resources` 旧字段的双重真相源 | P2-02 | 旧字段删除前所有值已迁移 |

### P3：完整观测、内容表达与个人外观

目标：扩展对结果的解释能力，并提高创作与个人使用效率。

| ID | 任务 | 依赖 | 验收 |

| P3-02 | 数据分析仅从遥测事件与 Outcome 派生 | P3-01 | 每项报表能追溯到事件范围 |
| P3-03 | 实现 `BlockCallout`、`BlockResource`、`BlockStep` | P2-01 | 每个新块遵守宽度契约 |
| P3-04 | 至少一篇真实公开文章使用新 Block | P3-03 | 不使用专用演示页代替真实使用 |

| P3-06 | 提供跨设备预览、保存视图、批量处理和差异预览 | P3-01, P3-05 | 不增加无来源装饰性指标 |

### P4：AI 自动化与 Contributor

目标：在真实契约和 Human Gate 稳定后提高吞吐量。

- AI 生成有 EvidenceRef 的决定提案、Action 草稿和 LearningRequest。
- Skill 自动生成、回放评测、有限注册、暂停与回滚。
- Contributor 只能读写被授权对象。
- 建立分配、审核、评论、操作日志与责任归属。

验收锚点：AI 和 Contributor 可以提高处理速度，但不能绕过 Walker 的权限、公开边界和高风险确认。

### P5：NorthStar 与经营系统

目标：在个人系统可独立运行后，增加可切换的公共网络与经营范围。

- 选择性发布 content / capability / offer / request。
- 可解释的需求、能力、人与项目匹配。
- 联系、协作和 Outcome 回流。
- 商品、服务、订单、支付、结算、退款与争议处理。
- 个人私密对象不自动同步到公共网络。

验收锚点：关闭 NorthStar 和经营范围后，Walker 个人系统仍可完整工作。

P0 真实性契约
  -> P1 反馈 API 与需求到结果闭环
      -> P2 内容规模化

## 8. 原计划冲突修正

### 8.1 禁止用 localStorage 作反馈降级真相源

`content-detail-page-todo.md` 中“API 不可写时先存 localStorage”与后台真实性原则冲突。修正为：

- 可以用 localStorage 防止同一浏览器重复点击。
- 不得把 localStorage 记录计入后台反馈或 hit-rate。
- 真实写入 API 未成立时，不上线可提交的反馈按钮。

### 8.2 明确反馈早于点赞

点赞只是弱信号。“有用 / 需补充 / 已过时”能直接导向行动，因此 `BlockFeedback` 和真实 API 早于 `BlockLike` 整合。

### 8.3 MDX 迁移不是独立目标

- 只有需要内容 Block 时才必须迁移为 MDX。
- 迁移不得早于真实反馈闭环。
- 不为了文件后缀统一而增加内容风险。

### 8.4 退役代码使用 Git 历史追溯

`ArticleLayout` 零引用且构建、视觉验证通过后直接删除，使用 Git 保留历史；不在 `src/_archive` 中继续保存死代码。

## 9. 各阶段统一验收

每个 Phase 都要通过：

1. **真实性**：数据有来源、时间、范围和更新状态。
2. **可操作性**：每项数据能导向判断、动作或明确结论。
3. **持久性**：业务状态不因刷新、换浏览器或重启丢失。
4. **空状态**：无数据时不生成假数字和假 AI 建议。
5. **错误状态**：明确告知失败模块、影响和可采取动作。
6. **权限**：未授权角色无法读写不属于它的对象。
7. **可访问性**：键盘、焦点、对比度和减少动效正常。
8. **回归**：类型检查、测试、构建和目标页浏览器验证通过。

## 11. 完成定义

本综合方案的第一个成立点不是“所有页面都重设计”，而是：

```text
选择一条真实 NeedCase 或 WalkerThesis
-> 打开原始证据
-> Walker 作出决定
-> 形成可追踪 Action
-> 产出或修改一篇内容
-> 发布到真实详情页
-> 收到至少一条真实明确反馈
-> 生成 Outcome
-> 根据 Outcome 形成下一次决定
```

只有这条链路成立，后台和内容详情页才算属于同一个产品系统。
