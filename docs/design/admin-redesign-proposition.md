# Walker 后台重设计 · 命题书

> 给命题，不给解法。方向对了，我们再往下走。

## 我看到的现状

现在的后台是一间赤陶色的会计室：白色底子、#c4704a 强调色、17 个入口平铺在一条横栏里、每个页面自己手写一套样式。它和前台那艘会发光的海船不像同一个生命体。

但 Pencil 里已经有一份探索稿（`Admin — Dashboard` 等帧），它做了两件对的事：

1. 把导航从横栏收进左侧，按「认知闭环 / 系统资产 / 运维」分组；
2. 把后台从"内容管理器"重新组织成"今天该处理什么"。

这份命题书不是推翻它，而是把它再往深处推一步：让它彻底属于 Walker 的航海隐喻，让它从"现代 SaaS 后台"变成一个"只服务于 AI 驱动个人创作系统"的驾驶舱。

---

## 核心命题：后台是一艘船的舰桥

前台的隐喻是 Ferry / 渡船 / 航海日志。后台应该是同一艘船的**舰桥（Bridge）**——不是展厅，是夜里掌舵的地方。

舰桥的心智模型：

- 黑暗中，仪表发着微光；
- 船长一眼扫过去，知道船在哪、哪块仪表在报警、下一步该往哪打舵；
- 每一个数据都指向一个动作，而不是用来满足好奇心。

对应到 Walker 的后台：

| 舰桥 | Walker 后台 |
|---|---|
| 罗盘 / 海图 | 内容宇宙、选题、需求地图 |
| 航海日志 | 访客需求记录与复盘 |
| 仪表板 | 今日要务、Gateway 健康、命中率 |
| 舵轮 | 内容编辑、简报生成、发布 |

所以后台不是"管理内容"，而是**让每一个数据信号都能被转化成一次航行调整**。

---

## 设计方向：Bridge at Night

### 一句话

一个深色、仪器感、以动作优先的舰桥界面：数据像海图上的标记，每个标记都可以被点击并变成一次内容或功能的调整。

### 三条不可让步的原则

1. **实用第一，好看第二**
   每一屏都要让作者知道"现在该做什么"。不是看数字，是动手改东西。

2. **同一艘船的不同舱室**
   后台必须和前台共享 DNA：#35bfab 翡翠青、深海暗底、玻璃与微光、航海命名。但不能照搬前台的浪漫飘逸，后台要更紧实、更工具化。

3. **拒绝通用后台模板**
   不要左侧深色导航 + 右侧白色内容卡片的标准 SaaS 格局。我们要的是"船在夜里"的整体氛围，而不是"后台换肤"。

## 视觉语言

### 色彩

- **背景**：深海 `#070c12`（`fullscreen-layout` 已验证过的颜色）
- **面板**：半透明船体玻璃 `rgba(10, 23, 36, 0.7)` + `backdrop-filter`
- **结构线**： helm-border `#1e3a45`，像船舱里的金属接缝
- **主光**：翡翠青 `#35bfab`，用于当前状态、可行动作、脉冲指示
- **辅助光**：青色 `#1fc9e7`、琥珀 `#f59e0b`、玫瑰 `#e11d48`
- **文字**：羊皮纸白 `#f1f5f9`、暗淡星光 `#94a3b8`、远处灯火 `#64748b`

### 字体

- 数据与状态：`JetBrains Mono`（等宽，像仪器读数）
- 标题与章节：`Averia Gruesa Libre`（前台 heading 字体，保持同一艘船）
- 正文：`Inter` / 中文系统字体

### 质感

- 玻璃面板带内发光与极细边框
- 背景 faint grid（ nautical chart 网格暗示）
- 脉冲点表示"需要关注"
- 不使用大阴影、大圆角、渐变装饰。克制，像仪器。

## 三条航线：3 个优先页面

不是 17 个平铺入口，而是 3 条核心工作流。每一条都回答"作者进后台后该做什么"。

---

### 页面一：舰桥总览 / Bridge Deck

**核心问题**：今天哪些事在排队、哪些仪表在报警？

**结构**：

- 左侧 Helm Rail（固定导航）
- 顶部 Instrument Bar：当前班次、在线访客、Gateway 状态、未读反馈
- 今日要务卡片：不是统计数字，而是"待分流 / 待改写 / 待立项 / 待裁决"
- 洋流 Currents：最近 24h 的实时信号流
- 罗盘 Quick Compass：四个最常用的下一步动作

**为什么这样**：作者一进来就知道今天该动哪块舵。每个卡片都有一个动词。

---

### 页面二：航海日志 / Logbook

**核心问题**：访客需求怎么变成内容动作？

**结构**：

- 左侧需求列表，按"待分流 / 待改写 / 待立项 / 已解决 / 已忽略"过滤
- 每条需求显示：来源、匹配到的内容、反馈状态
- 每条需求有三个动作按钮（例如：补问答 / 忽略 / 已解决）
- 右侧航线 Route：把"需求 → 匹配 → 反馈 → 动作"可视化
- 航线下方 Composer：AI 或规则给出的动作建议，一键生成简报 / 编辑 / 入池

**为什么这样**：把 `NeedCase` 这个核心业务对象做成了可操作的日志条目。作者不再是在看访客留言，而是在"批改航线"。

---

### 页面三：海图台 / Chart Table

**核心问题**：怎么把信号和决策变成一篇内容？

**结构**：

- 左：Signal Board（需求簇 / 热度信号）
- 中：Brief Card（问题、角度、结构、关联、验证方式）
- 右：Draft Editor（深色编辑器，Markdown，自动保存）

**为什么这样**：把"选题 → 简报 → 草稿"放在一个空间里完成。海图台的隐喻是一张桌子：左边摊着地图（信号），中间是罗盘和尺规（简报），右边是正在画的航线（草稿）。发布按钮不是"发表"，而是"发布并关联需求"——闭环在这里闭合。

## 信息架构重塑

现有 17 个页面要被收进舰桥的四个舱室：

```
值班（每天进后台的第一件事）
  ├─ 舰桥总览
  ├─ 航海日志
  └─ 海图台

观测（周期性查看，验证效果）
  ├─ 客流与行为
  ├─ 内容命中率
  └─ 需求复盘

机舱（系统健康与规则）
  ├─ AI Gateway
  ├─ 规则池
  ├─ Skill 候选
  └─ 安全事件

船员（账号与权限）
  ├─ 账号
  └─ 邀请码
```

入口从 17 个降到 13 个，更重要的是它们被组织进了**工作流**，而不是 CRUD 实体。

## 与前台的关系

前台是访客看到的船：明亮、流动、有雾气与光球。后台是船长所在的舰桥：暗、紧凑、所有光都指向可操作的仪表。

两者共享：

- `#35bfab` 翡翠青作为系统 DNA
- 航海命名（Helm、Currents、Route、Chart Table）
- Averia Gruesa Libre 作为标题字体
- 玻璃质感（前台是晨雾玻璃，后台是仪器玻璃）

两者不同：

- 前台情绪化、探索性；后台动作化、决策性
- 前台浅色为主；后台深色为主
- 前台信息密度低；后台信息密度高但分区清晰

## 惊喜点在哪里

1. **"今日要务"取代"统计仪表盘"**
   不是"内容总数 47"，而是"3 篇内容该改写、1 个功能该裁决"。每个数字都是动词。

2. **航海日志即工作流**
   需求不再是一堆消息，而是可以被"分流 / 改写 / 立项 / 忽略 / 已解决"的日志条目。右侧的 Route 可视化让闭环一目了然。

3. **海图台把创作三段式摊开**
   信号、简报、草稿在同一屏。不是跳三个页面，而是在一张桌子上同时看到"为什么写、写什么、怎么写"。

4. **整体暗色 instrument UI**
   不是"深色模式"，而是"舰桥在夜里"。这种氛围在中文个人站点的后台里几乎没有先例。

## 我想问你的

方向上有三个可选的深浅：

- **A. 保守**：沿用现有 Pencil light-direction 稿，只优化信息架构与动作流；
- **B. 中等**：整体深色，但保留较多卡片与列表的 SaaS 可读性；
- **C. 激进**：彻底舰桥化，像仪器面板一样紧凑、暗色、航海隐喻贯穿每个细节。

我这次给的是 **C（激进）**。如果你也觉得对，我们再往下走：细化组件、出剩余页面、讨论哪些现有 admin 页面可以合并或删除。

如果你要调方向，告诉我往 A/B 哪边走，或者哪个细节让你不舒服。
