# Walker 后台重设计 · 交付规范

> 这是标准，不是建议。照此 1:1 还原。
> 配套设计源文件：`admin-redesign.pen`（所有画板、组件、变量的权威来源）。
> 遇到本规范与 `.pen` 文件冲突，以 `.pen` 为准。

---

## 0. 项目一句话

Walker 是一个 AI 驱动的个人 Agent 系统网站。访客从自媒体引流来，看文章、提问、被 Agent 匹配。后台是只给作者和 contributor 看的驾驶舱，核心使命是让「访客需求 → 内容产出 → 验证效果 → 再调整」这条闭环跑起来。

## 1. 设计系统

### 1.1 配色（写死，不要改）

**前台 DNA（前后台共用）：**

| 用途 | 色值 | 说明 |
|---|---|---|
| `brand` | `#35bfab` | 翡翠青。主强调色，前台后台统一 |
| `brand-dim` | `#1a8f80` | 翡翠青深色，渐变/边界 |
| `brand-soft` | `#35bfab1a` | 翡翠青 10% 透明，标签底色 |

**主区（亮色）：**

| 用途 | 色值 |
|---|---|
| `bg` | `#fbfbf8` 主区背景（奶白） |
| `surface` | `#ffffff` 卡片/顶栏底 |
| `text` | `#1a2325` 正文 |
| `text-dim` | `#64748b` 次要文字 |
| `text-faint` | `#94a3b8` 占位/极次要 |
| `border` | `#e7e5e0` 卡片/分割线 |
| `border-soft` | `#f0efeb` 内嵌浅分割 |

**侧栏（深色）：**

| 用途 | 色值 |
|---|---|
| `sidebar-bg` | `#0f1a1c` 侧栏底（深炭青） |
| `sidebar-surface` | `#16282b` 身份区/底部底 |
| `sidebar-text` | `#c8d4d6` 导航文字 |
| `sidebar-text-dim` | `#6b8488` 组标签/次要 |
| `sidebar-active` | `#35bfab` 激活态（= brand） |

**状态色（全局语义，必须一致）：**

| 语义 | 主色 | 软底色 | 用于 |
|---|---|---|---|
| 好/正常/成功 | `#35bfab`（brand） | `#35bfab1a` | 公开、成功、活跃、健康 |
| 观察/偏低/草稿/警告 | `#f59e0b`（amber） | `#f59e0b1a` | 草稿、偏低、偏高、降级、接近预算 |
| 问题/失败/危险/冷门 | `#e11d48`（rose） | `#e11d481a` | 失败、命中下降、冷门功能、熔断、降级 banner |
| 中性/私密/无效 | `#94a3b8` | `#94a3b81a` | 私密、无效、非激活 |
| 辅助类型色 | `#3b82f6`（工具）/ `#a855f7`（随笔）/ `#ec4899`（生活） | 对应 `xx1a` | 内容类型标签 |

### 1.2 字体

- **UI 字体**：`Inter`（所有界面文字）
- **数字/时间码**：`JetBrains Mono`（指标数值、时间戳、延迟等技术数字）

### 1.3 图标

- **库**：`lucide`（统一用这个，不要混用其他库，不要用 emoji）
- **尺寸**：导航/行内 14-16px，按钮内 13-14px，大图标 30px

### 1.4 尺寸与圆角约定

- 画板：1440×900（桌面基准）
- 侧栏宽：248px
- 主区内边距：横向 32px，纵向 18-24px
- 卡片圆角：10-12px；按钮/标签 6-8px；徽章 999（胶囊）
- 触控目标最小：42px 高（navItem 已是这个值，落代码时按 44px）

### 1.5 状态语义铁律

- **每个数据都要带趋势/基准**（环比箭头 ↑↓ + 百分比 + 颜色）。孤岛数字不被接受。
- **每个展示项都要通向一个动作**（按钮/链接）。纯展示不被接受。
- **色盲安全**：状态不能只靠颜色，要同时带文字标签或图标。

## 2. 共享骨架 AdminLayout

所有页面共享这个骨架。**先建这个，再填页面。**

```
┌─────────────────────────────────────────────────┐
│ 侧栏 248px  │            主区 fill               │
│ (深色)      │  (亮色)                            │
│             │                                     │
│ ① 身份区    │  ┌─ 顶栏（页面标题 + 副信息 + 操作）│
│ ② 导航区    │  ├─ [可选] 降级 banner / 优先级条   │
│   (三组折叠)│  ├─ [可选] 筛选行 / tab 栏          │
│ ③ 底部      │  └─ 内容区                          │
└─────────────────────────────────────────────────┘
```

### 2.1 侧栏

**① 身份区**（`sidebar-surface` 底）：
- 圆形头像 40×40（翡翠青渐变填充，未来换成作者图）
- 名字（14px / 700 / 白）+ 角色（11px / brand：「Owner · 驾驶舱」）
- padding 20，gap 12

**② 导航区**（`sidebar-bg` 底，fill 撑高）：
- 三个可折叠组，每组 = GroupHead + Children 容器
- **分组与顺序（从上到下，= 工作流顺序）：**
  1. **认知闭环**（4 项）：洞察 / 选题库 / 命中率 / 需求复盘
  2. **系统资产**（4 项）：内容管理 / 规则池 / 经验 / Skill 候选
  3. **运维**（4 项）：AI Gateway / 用户 / 邀请码 / 事件
- 默认展开「认知闭环」，其余收起
- 见 §3 组件：NavItem、GroupHead

**③ 底部**（`sidebar-surface` 底）：
- 「查看站点」（external-link 图标）/ 「登出」（log-out 图标），`sidebar-text-dim` 色

### 2.2 NavItem 激活态

- 激活项：图标 + 文字变白、文字加粗（600）
- 非激活：`sidebar-text`（#c8d4d6）
- 计数徽章：右侧，正常用 `sidebar-text-dim`，有待办用 amber/rose/brand

## 4. 逐页规范

### 4.1 工作台（Dashboard · `/admin`）

**这是首页。进来第一眼看到"今天该干啥"。**

**布局：**
- 顶栏：标题「今日工作台」+ 日期（2026年6月18日 · 周四）+ 右侧「新建创作」按钮（brand 底 + plus 图标）
- **降级 banner**（条件渲染，Gateway 异常时显示）：深红 `#7f1d1d` 底，alert 图标 + 主文案「AI Gateway 昨日降级 2 次，3 次调用失败」+ 副文案 + 「查看事件」按钮（白底红字）+ 关闭叉
- 内容区：
  - **4 个 Metric 卡**（横向排列）：本周需求 47（↑12 较上周·brand）/ 内容命中率 68%（稳定）/ 待立项选题 23（需决策·amber）/ Gateway 健康 92%（昨日降级 2 次·rose）
  - **待处理需求列表**（主角）：标题「今天该处理的需求」+ 副信息「按闭环阶段排序 · 共 5 条」+ 右侧筛选按钮
  - 5 条 NeedRow，按闭环阶段排序，阶段标签颜色编码：
    - 🟢 待立项 ×2（brand soft）→ 动作「立项」
    - 🟡 写作中（amber soft）→ 「继续」
    - 🔵 待验证（蓝 `#dbeafe` / `#2563eb`）→ 「复盘」
    - 🔴 命中降（rose soft）→ 「查看」

**数据字段（NeedRow）：**
- 阶段标签文字 + 颜色
- 需求标题（用户问什么）
- meta：「N 人问 · 卡点：xxx · N 天前」/「已立项 · 草稿进行中 · 写了 60%」/「命中率 41% → 28% · 近 7 天 N 条 stuck」
- 动作按钮文字

### 4.2 工作台 · 空状态

当本周没有待处理需求时显示。

- 居中大卡（520 宽，白底）
- 翡翠青圆圈 + inbox 图标（30px）
- 标题「本周还没有待处理的需求」
- 引导文案（居中、1.6 行高）：「需求会随访客提问自动汇聚到这里。现在可以去洞察看看大盘，或去选题库手动添加一个你想写的方向。」
- 两个按钮：**去洞察看大盘**（brand 实底，主）/ **去选题库**（灰边，次）

### 4.3 洞察（Insights · `/admin/insights`）

**数据页模板。三个 tab，顶部有 AI 优先级行动条。每个 tab 都通向"改"的出口。**

**顶栏：**「洞察」+「看数据 → 改内容 / 改功能 · 近 30 天」+ 右侧范围选择器（「近 30 天」+ chevron-down）

**「本周该做的 3 件事」（AI 建议条，三个 tab 之上）：**
- 左侧 lead：「AI · 本周建议」+「本周该做的 3 件事」
- 右侧 3 张优先级卡（横向），按严重度排序：
  1. 🔴 改提问匹配降级（流失 61%↑ · 本周最该修）— 红序号 `#e11d48`
  2. 🟢 写 P0→P2 完整案例（18 条反馈在等 · 接 Ferry 流量）— brand 序号
  3. 🟡 换学习轨道标题再试（14 看或因引流弱 · 先挽救再删）— amber 序号

**Tab 栏：**「行为 · 改功能」（激活，下划线）/「内容 · 改文章」/「反馈 · AI 归类」

#### Tab 1 · 行为（改功能）

两列布局：

**左列「功能使用热度」：**
- 标题 + 副信息「次数低 = 考虑删功能 · 次数高 = 加固它」
- 6 条 FeatureRow，按使用次数降序，**每条带环比**：
  - 提问匹配 482（活跃·brand）↑18% → 「加固」
  - 文章阅读 391（活跃）↑9% → 「加固」
  - 学习轨道 74（观察·amber）↓4% → 「观察」
  - 内容关系图谱 38（偏低·amber）↓12% → 「观察」
  - 点赞 9（冷门·rose）↓41% → 「考虑删除」（rose 软底按钮）
  - 项目展示 3（冷门·rose）↓63% → 「考虑删除」
- 每条：名称 + 状态标签 + 使用条（颜色=状态色，宽度=按次数）+ 次数+环比（环比染色）+ 动作按钮

**右列「访客从哪来 · 在哪走」：**
- 来源卡：自媒体公众号 1240（bar brand）/ 直接访问 312 / 搜索引擎 88
- 流失环节卡（带「需改善」rose 标）：
  - 提问后无匹配走了 61% ↑（**均值 48% · 本周异常**·rose）→「改匹配」
  - 进首页直接走 44% ↑（均值 39% · 略升·amber）→「改首屏」
  - 文章读到一半走 23% →（均值 24% · 持平·灰）→「改内容」

#### Tab 2 · 内容（改文章）

文章热度榜，5 条 ArticleRow，按热度降序：
- 每条：排名 + 标题 + 「看 N · 问 N · 停留 Nm · ↑/↓ N%」+ 动作
  - Ferry 隐喻（看842·↑22%）→「写更多」（brand）
  - P0-P4 方法（看531·↑11%）→「写更多」
  - AI Gateway 匹配（看318·问88·停留2.4m·↓8%）→「补问答」（amber）
  - 双 Git 协同（看96·→持平）→「改写」（amber）
  - 学习轨道（看14·↓35%）→「藏起来」（rose）

#### Tab 3 · 反馈（AI 归类）

**不是计数，是聚类。** AI 把反馈聚成主题，每主题取代表原话 + 建议 + 动作。

- 标题「访客反馈 · AI 已聚类」+ 副信息「37 条反馈 · AI 聚成 4 个主题 · 每主题取代表原话 + 建议」
- 4 张 ThemeCard（不是 37 张散卡）：
  1. **P0→P2 完整案例怎么做**（该写教程·18 条·置信 92%）— 代表原话 + AI 建议「写一篇承接 Ferry 流量，可拆连载」→「加入选题」（brand）
  2. **提问匹配不上就白屏**（该改 bug·9 条·置信 88%·**与流失率 61% 交叉印证**）→「去改」（amber）
  3. **双 Git 协同看不懂**（该补素材·5 条·置信 79%）→「去补」（brand）
  4. **泛泛抱怨**（无效吐槽·5 条·置信 64%·未指向具体内容）→「全部忽略」（灰）

**ThemeCard 结构：** 头部（标签 + 主题名 + 计数胶囊）/ 代表原话区（引号 + 原话 + 「另有 N 条相似反馈 · 置信 N%」）/ AI 建议条（sparkles 图标 + 建议 + 动作按钮，整条带 soft 底色）

## 5. 交互行为（静态稿画不出，落代码必须实现）

| 行为 | 说明 |
|---|---|
| **侧栏折叠** | 点 GroupHead 切换 chevron down↔right，Children 容器显隐。默认仅「认知闭环」展开 |
| **Tab 切换** | 洞察页三 tab，点切换激活态（下划线 + 加粗），内容区切换 |
| **降级 banner** | Gateway 异常时条件渲染，可关闭 |
| **动作按钮跳转** | 所有「立项/去写/去改/加入选题/查看事件」按钮跳到对应页或打开编辑器 |
| **hover/focus** | 所有可点元素 hover 有反馈（边框/底色变化），focus 有可见 ring |
| **键盘** | Tab 顺序 = 视觉顺序；侧栏组可 Enter/Space 折叠（aria-expanded） |
| **骨架屏** | 数据未回时显示骨架，不空白跳动 |

## 6. 暂不做 / 留位 / 未来

明确不在本规范范围，避免过度设计：

| 项 | 状态 | 原因 |
|---|---|---|
| Agent 治理页 | **侧栏留位**（灰色「敬请期待」组） | Agent 自主调用还没落地，无数据。架构预埋，未来填 |
| 操作日志/审计 | 暂不做 | 单用户不需要；有 contributor 后补 |
| RBAC 权限 | 暂不做 | 邀请制 + 硬编码 admin 够用；团队大了再说 |
| 移动端 | 暂不做 | 先桌面稳；移动端侧栏改抽屉式 |
| 归因/漏斗/情绪分 | 暂不做 | 需要更多数据源；现在做是空架子 |
| 电商/营销/订单 | **永不做** | 跟本项目物种不同 |

## 7. 如何使用本规范

1. **先看 `.pen` 文件**——所有画板的真实视觉在那里，本规范是它的文字版说明
2. **先建 AdminLayout 骨架**（§2）——所有页面共用
3. **再建组件**（§3）——建成可复用的框架组件
4. **逐页填充**（§4）——按规范的数据字段和状态
5. **实现交互**（§5）——静态稿画不出的行为
6. **不要发挥**——这是 B 模式交付，1:1 还原。配色/布局/组件/状态都写死了。改动需先和作者确认。
