# 后端架构与技术栈演进方案

> **日期**:2026-06-22
> **状态**:部分实施 — WorkItem domain 剥离已完成（2026-06-22, commit 0ba430a→ff83f35）；PG 迁移、middleware、可观测性待开始
> **定位**:本文件是技术选型与架构演进的**候选设计输入**,不拥有当前执行权。执行三件套在 `.agents/skills/walker-northstar/references/working/`。
> **北极星**:长期承载**数十万人同时访问**的内容 + AI 平台。
> **方法**:演进式六边形——现在轻装起步,把可移植的地基打好,终态收敛到 Go 核心 + Node BFF 的混合云原生架构。

---

## 一、背景与目标

### 1.1 现状(Walker 数字花园)

- **形态**:TypeScript 全栈。Astro 6 SSR(`output: 'server'`)部署 Vercel serverless;四层架构 API → services → stores → conversation。
- **持久层**:21 个聚合根(WorkItem / NeedCase / Account / Session / Grant / Asset ...)全部堆在 Upstash Redis,`conversation/store.ts`(1940 行 / ~80 函数)手写 schema + 索引 + MULTI 事务 + 内存兜底。
- **已诊断的结构性问题**(详见附录 A,均有代码证据):
  - Port+Adapter 分层失灵:9 个 `*.store.ts` 是薄壳转发,Port 与 store 是"同一能力两套名字",改 schema 要三处同步。
  - 降级语义双轨:5 个 service 走 `storage-mode` 503 拦截,16 条写入路径静默落内存(生产也降级)。
  - WorkItem 状态机三个真实 bug(审计失真 / 过期 proposal 回流 / canTransition 不一致)。
  - 路由层 71 处 `isAdmin/isOwner` 散落 45 个文件,无 middleware。
  - admin 页 SSR 了壳却用客户端 fetch 二次取数据(联调类型断裂)。
- **已做对、需保留的部分**:`callGateway`(真深模块)、`WorkbenchService` 写路径 + `mutate` 不可变、`ContentFileStore`(GitHub/本地 git 策略对等)、`like.store`(incr 原子 + 独立双层降级)、`storage-mode` 三态护栏、`requireHighRiskAudit`(fail-closed)、内容 git 持久(天然高可用)。

### 1.2 目标与核心张力

- **北极星**:数十万人同时访问。这是**质变级**目标,把项目从"个人数字花园"赛道切到"互联网产品"赛道,伴随团队 / 成本 / 运维的量级跃升。
- **核心张力**:现状是个人项目阶段(流量未到),但长期目标要求互联网产品级架构。
- **解法**:不为"未来可能"的规模提前买单,但把**任何终态都不浪费的地基**现在就打好——演进式,不推倒重来。

## 二、决策摘要

**一句话**:以"演进式六边形"为骨架——现在用 TypeScript on Node 轻装起步,把可移植的地基(domain + PostgreSQL + 可观测性)打好,终态收敛到 **Go 核心 + Node BFF** 的混合云原生架构。语言会换,地基不浪费。

**关键决断**:

1. **数据库**:PostgreSQL(已定)。用 **Drizzle** 接入(轻、SQL 透明、serverless 冷启动友好)。Redis 退守 KV 强项(incr / 限流 / 短 TTL 会话 / 缓存)。
2. **核心后端语言**:现在 **TypeScript on Node**(演进最佳起点:开发效率 + AI/MCP 生态最前沿 + domain 可移植);终态 **Go**(资源效率 + goroutine 扛并发 AI + 云原生)。**不为"已写 TS"留 TS,而是因为它是最佳起点。**
3. **Java 不选**:治理能力对当前/中期规模过剩,成维护负担。仅在未来"接轨企业栈 + 团队扩张 + 强运行时治理"时 reconsider。
4. **架构形态**:六边形(`domain/` 纯逻辑 + `adapters/`),贯穿所有阶段,是"可演进"的根基。
5. **部署**:现在 Vercel serverless,并发破万迁 Kubernetes。

## 三、终态架构(北极星)

```
┌─────────────────────────────────────────────────────────┐
│  CDN + 静态预渲染        ← 几十万"读"的主要承载         │
├─────────────────────────────────────────────────────────┤
│  Node/TS  BFF + AI 网关 + MCP   (AI 生态 / 同构主场)    │
├─────────────────────────────────────────────────────────┤
│  Go  核心数据后端                (扛并发读写的核心)     │
│     ├─ 决策系统(WorkItem / 资产 / 授权)                │
│     ├─ 匹配 / Agent 编排(goroutine 并发 AI)            │
│     └─ 账号 / 会话                                       │
├─────────────────────────────────────────────────────────┤
│  domain/  纯逻辑层(从 TS 移植,语言无关)              │
├─────────────────────────────────────────────────────────┤
│  PostgreSQL(主从 / 读写分离 / Citus 分片)              │
│  Redis(缓存 / 限流 / incr / 会话短 TTL)                │
│  Kafka / NATS(异步削峰,AI 异步化)                     │
├─────────────────────────────────────────────────────────┤
│  Kubernetes + Prometheus / Grafana / OTel + 多可用区    │
└─────────────────────────────────────────────────────────┘
```

**为什么不直接一步到终态**:现在就上 K8s + Go + 分片 = 用巨额复杂度和成本买**当前用不到**的指标。演进式的价值在于:现在轻装,但每一块都是未来终态的零件,不推倒重来。

## 五、架构形态:六边形(可演进的根基)

贯穿所有阶段,不变的是这个形态:

```
domain/                 纯 TS,零 IO 依赖,可纯单测覆盖
  workitem/             (状态机 + 不变量)
  matching/             (匹配规则 + Agent 编排逻辑)
  assets/               (资产生命周期规则)
  ports/                (持久化 / 时钟 / AI / 限流 是接口)

adapters/
  pg/                   Drizzle schema + repository → 实现 domain port
  redis/                incr / 限流 / 缓存
  http/                 Astro endpoints → 调 domain,做协议映射
  mcp/                  MCP server → 调 domain(HTTP / MCP 共享领域逻辑)
  ai-gateway/           AI provider 调用 + 降级
```

**为什么这是根基**:`domain/` 不知道 PG / Redis / HTTP 的存在。未来切 Go 时,domain 是移植蓝本(纯逻辑、无运行时依赖),Go 只重写 adapter;留 TS 时,domain 就是六边形核心。**无论终态语言为何,domain 这层的设计与测试都不浪费。** 这是"基于现状"能做的最值钱的提前投入,也是修上一轮"Port 薄壳失灵"的根治方案——把领域逻辑从"Astro 请求 + Redis 调用 + 业务规则"三件事耦合中剥离。

## 六、五阶段演进路线图

| 阶段 | 做什么 | 为什么 | 触发线 | 产出 |
|---|---|---|---|---|
| **1. 地基**(现在) | domain 剥离 + PG 迁移 + middleware + 可观测性 + 修 WorkItem bug + DTO 共享 | 任何未来路径的必经之地,不浪费 | 立即 | 可移植领域核心 + PG + 观测 |
| **2. 读优化** | admin 读挪 SSR props + CDN 强化 + Redis 缓存层 + 限流强化 | 卸读压力到 CDN / 缓存,后端减负 | 日 PV 破十万 | 读路径几乎不打后端 |
| **3. 引入 Go** | 把匹配 / AI 服务抽成独立 Go 服务,Node 退居 BFF / MCP | 首次扛并发瓶颈;goroutine 并发 AI | 并发破万 | 混合栈起步 |
| **4. 分布式** | 迁 K8s + PG 读写分离 / 读副本 + 熔断 / 限流 / 降级全套 | serverless 成本爆炸 + 单点瓶颈 | 并发破十万 | 自治可扩缩 |
| **5. 分片** | PG 分片(Citus)+ 多可用区 + 异步削峰(Kafka) | 真·互联网产品形态 | 并发破几十万 | 数十万并发承载力 |

**纪律:每阶段只解决当时的真实瓶颈,不为下一阶段提前买单。触发线没到就不动。**

### 规模校准(避免误判)

| 说法 | 工程含义 | 真实参照 |
|---|---|---|
| 几十万注册用户 | 几乎没压力 | 任何栈能扛 |
| 几十万日活(DAU) | 峰值并发约几千~1万 | 中型博客 / 工具站 |

## 七、第一步执行清单(阶段 1,2-3 周,按优先级)

1. **domain 剥离 + 修状态机 bug**(最高杠杆)
   - 把 `workbench.service` 的 WorkItem 状态机抽成纯 TS `domain/workitem/`,零 IO,纯单测锁住不变量。
   - 修附录 A.2 的三个 bug:`updateAction` 审计 fromStatus 执行序 bug、`overridePriority` 自反审计、`list()` 不过滤过期 proposal、canTransition 不一致。
   - **产出**:领域逻辑可被纯单测覆盖,为未来移植 Go 备好蓝本。

2. **PostgreSQL 迁移(核心域)**
   - Drizzle 定义 WorkItem + Action + Outcome + Evidence schema + 迁移。
   - `adapters/pg/` 实现 domain port,与 Redis **双写过渡**,读切 PG,验证后下掉 Redis 的领域状态职责。
   - **产出**:消灭 god store 手写 schema;`getNeedCaseStats` 76 行扫描 → 一句 SQL。

3. **middleware 集中鉴权**
   - `src/middleware.ts` 统一 `/admin/*` + `/api/admin/*` 鉴权,71 处 `isAdmin` → 1 处。
   - **产出**:消除路由层最大的 change amplification,顺带给附录 A.4(高权写校验 session)落地位置。

4. **可观测性地基**
   - Sentry(错误)+ 结构化日志(pino)+ 关键业务指标埋点(匹配成功率 / AI 降级率 / DB 延迟)。
   - **产出**:这是判断"该不该进下一阶段"的依据——没有数据,演进触发线无从谈起。

5. **共享 DTO + 类型 client**
   - API 请求 / 响应类型抽到 `shared/dto/`,前后端 import 同一个 type;`apiFetch<T>` 泛型 client。
   - **产出**:消除前后端类型断裂,65 处裸 fetch 收口。

## 八、不变量——无论未来怎么走都不浪费的投入

| 投入 | 为何不浪费 |
|---|---|
| **domain 纯 TS + 单测** | 移植 Go 的蓝本;留 TS 的核心。语言无关 |
| **PostgreSQL schema** | 切 Go 时 schema 不变,只换 client(Drizzle → sqlc) |
| **middleware 鉴权** | 任何语言 / 框架都要做 |
| **可观测性** | 规模越大越值钱,从第一天就有用 |
| **Redis 退守 KV 强项** | 从阶段 1 到阶段 5,职责定义不变 |

**这五样是"基于现状"唯一值得现在就投入的提前架构。** 其余(Go / K8s / 分片 / 异步队列)等真实流量触发,不提前买单。

## 九、决断理由

1. **接受北极星(数十万并发)** → 终态必须是 Go 核心 + Node BFF 的混合云原生栈,单一语言扛不住。
2. **尊重现状(个人项目、流量未到)** → 现在不一步到位,演进式,避免用复杂度买当前用不到的指标。
3. **排除沉没成本但务实** → 现在留 TS 不是因为"已写",而是它是演进的最佳起点(开发效率 + AI/MCP 最前沿 + domain 可移植);切 Go 是明确的终态,不是"可能"。
4. **修真问题** → 第一步直接修上一轮评审发现的 WorkItem 状态机 bug、god store、降级双轨、71 处重复鉴权——不是空谈架构。
5. **Java 不选的边界** → 治理能力过剩;仅在未来"接轨企业栈 + 团队扩张 + 强运行时治理"时 reconsider。

---

## 十、诚实的警告

**先验证产品到真实高流量,再为高流量做架构。** 90% 预期"未来要扛几十万并发"的项目从没到过那个量。提前全套分布式架构的成本(团队 / 服务器 / 运维 / 复杂度)是实打实付出去的,而"未来几十万并发"的收益是可能永远不兑现的。

**正确的长期价值不是"现在就搭一个能扛几十万并发的架构",而是"现在搭一个能平滑演进到几十万并发的架构"。** 本方案即此后者。

## 附录 A:本方案回应的关键问题(架构评审发现,均有代码证据)

### A.1 存储分层失灵(P1,High)
- `conversation/store.ts` 1940 行 / 21 聚合根 / ~80 函数,god module。
- 9 个 `*.store.ts`(need-case / feedback / work-item / ...)是纯薄壳转发(亲验 `need-case.store.ts` 全文)。
- Port 与 store 是"同一能力两套名字",改 schema 要 ports.ts / store.ts / *.store.ts 三处同步。
- 7 个 RepositoryPort(ObjectGrant / ActionAudit / NorthStar / SupportConfig / Experience / Rule / Skill)有 Port 定义却无 store 实现,service 直接 import store 自由函数,绕过 Port。

### A.2 WorkItem 状态机三个真实 bug(P1,High,已坏)
1. `workbench.service.ts:351-358`——`updateAction` 审计 fromStatus 执行序 bug:`item.status` 先被改成 `awaiting-verification`,三元恒为 `'acting'`,history 违反"from=真实源状态"不变量。
2. `workbench.service.ts:444-451`——`overridePriority` 自反审计:`fromStatus===toStatus`(状态未变)却标 `kind:'status-change'`,污染状态历史投影。
3. `workbench.service.ts:460` vs `:464`——`isProposalExpired` 只在 `getTodayProjection` 用,`list()` 不过滤,过期 AI proposal 在 `view=decisions` 回流。
4. `canTransition` 白名单不一致:`decide`/`requestDecision`/`createProposal` 用,`createAction` 硬编码,`updateAction`/`recordOutcome` 完全不用——状态机不变量只守护一半迁移。

### A.3 降级语义双轨(P1,High)
- `getRedis()`(`store.ts:89-104`)只看 env、不调 `resolveStorageMode`、无环境区分(亲验)。
- 5 个 service(workbench / content-feedback / asset / content-telemetry / appearance)写前 503 拦截;16 条写入路径(NeedCase / MatchSession / ...)静默落内存。
- `NeedCase`(核心业务对象,admin-review 复盘依赖)与 `WorkItem` 护域相反。
- `getRedis` 被 17 处 import(Grep 实证),含 account / session / admin-audit——认证可用性绑在 conversation 模块,依赖方向倒置。

### A.4 认证会话失效窗口(P2,Medium)
- `admin-auth.ts:10-13`——`isAdmin` 纯 cookie 同步,不查 session store。
- `logout` / `killAllByUsername` 只删 Redis,旧 cookie 30 天 `SESSION_MAX_AGE` 内签名仍有效 → 封禁后仍可操作。
- 解法:高权写操作(删账号 / 改角色 / 删内容)额外校验 session store 存在性,普通读保持 cookie-only。

### A.5 路由重复鉴权(P1,High)
- 无 `src/middleware.ts`(Glob 实证)。`isAdmin/isOwner(request)` 在 45 个 API 文件散落 71 处(Grep 实证)。

### A.6 前后端联调断裂(P2,Medium)
- 客户端 `fetch('/api/')` 共 65 处 / 22 文件(Grep 实证)。
- admin 页 SSR 了壳却客户端 fetch 二次取数据(`admin/index.astro` 6 处、`topics.astro` 7 处)。
- 前端 `tool-match-view.ts` 有自己的类型,与后端 `MatchResult` 同源性未确认——大概率不共享 DTO。

### A.7 死代码抽象(P2,Medium)
- `object-authz.ts` 的 `canPerformObjectAction` / `authorizeAndAudit` / `canAccessAdminResource` 在 `/api/` 零调用(仅 grants 注释提及,Grep 实证)——Contributor 授权体系是 dead code。
- `createMatchingService()` 只被自身测试引用,生产匹配绕过它直接调 `matchSiteResources`(Grep 实证)。

### A.8 值得保留的设计
- `callGateway<T>`(真深模块,泛型 + fallback + parseResponse)。
- `WorkbenchService` 写路径(7 变更方法集中信任规则 + `mutate` 深拷贝 + `appendHistory` 单点)。
- `ContentFileStore`(GitHub API / 本地 git 策略对等实现)。
- `like.store.ts`(incr 原子 + 独立双层降级,完全独立于 god store)。
- `storage-mode.ts`(三态护栏,设计本身干净)。
- `requireHighRiskAudit`(fail-closed)。
- 内容 git 持久(天然高可用)。
