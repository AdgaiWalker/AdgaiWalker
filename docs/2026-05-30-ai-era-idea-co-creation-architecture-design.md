日期：2026-05-30

## 1. 时代定位

### 1.1 一句话定位

这是一个 AI 时代的点子共创基础设施：以个人系统保护人的主体性，以 AI 执行降低实现门槛，以点子连接人与人，在共同改造现实的过程中统一个人价值与社会价值。

更诗性的表达：

> 在 AI 的浪潮里，让每个人不被浪推走，而是找到自己的北星；用点子驱动 AI，用系统连接同行者，把个人的前进汇入共同的世界改造。

### 1.2 时代问题

AI 带来的不是单纯效率提升，而是执行能力的普及。过去普通人的很多点子死在不会写代码、不会设计、不会调研、不会表达、不会组织项目。AI 正在降低这些执行门槛。

因此，未来真正稀缺的不只是“能不能做”，而是：

- 为什么做
- 做什么
- 如何判断
- 对谁负责
- 如何持续
- 如何与他人协作
- 做完如何沉淀

本系统不是“又一个 AI 工具站”，而是 AI 时代普通人保持主体性、组织自身、连接他人、共同实现点子的社会基础设施雏形。

### 1.3 回应的时代矛盾

| 时代矛盾 | 产品回应 |
| --- | --- |
| 执行能力普及，但判断能力稀缺 | 人决策、AI 执行；AI 提供选项，人承担判断与责任 |
| 信息爆炸，但秩序稀缺 | 把混乱转成结构，把想法转成点子，把点子转成行动 |
| 个体能力被放大，但更容易迷失 | 用个人系统组织目标、内容、工具、反馈和复盘 |
| 人人可以表达，但很少共同实现 | 用点子把“我想说”推进到“我想做”和“我们一起做” |

## 2. 演化协议层：FerrySpec

### 2.1 FerrySpec 的定位

FerrySpec 是演化中的世界协议，不是静态教条。它提供当前有效的人机协作方法论内核，同时接受 iwalk.pro 和 NorthStar 的实践反哺。

FerrySpec 当前提供：

- 人类主权：人负责价值判断、审美判断和最终拍板；AI 负责逻辑推演、信息整理和方案生成。
- 熵减命令：将混乱、未知、情绪转化为有序、已知、可行动的结构。
- 蓝图 / 现实 / 日志：用 blueprint 表示应然，用 reality 表示实然，用 log 表示历史、复盘和进化。
- P0-P4 状态调度：混沌、规划、执行、修正、进化。
- Skill / Flow / Log：规则、执行、记忆的三位一体系统。
- AI 责任边界：AI 无法承担后果，不可逆操作必须由人确认。
- 做减法：如无必要，勿增实体。

### 2.2 三系统共同进化关系

```text
时代潮流
  ↓
FerrySpec 当前协议
  ↓ 指导
-iwalk.pro 个人实践场
-NorthStar 社会实践场
  ↓ 产生
阻滞 / 偏差 / 数据 / 用户反馈 / 社会影响
  ↓ 反哺
FerrySpec vNext
  ↓
继续指导下一轮实践
```

最终关系：

> Ferry 是演化中的协议，iwalk.pro 是个人实践场，NorthStar 是社会实践场；三者通过反馈、阻滞与复盘共同进化。

### 2.3 不把 Ferry 写成固定理论

本文档不把 FerrySpec 视为已经完成的“上位真理”。FerrySpec 当前指导实践，但真正的协议必须在 iwalk.pro 和 NorthStar 的持续实践中被检验、修正和扩展。

## 3. 价值宪法

以下原则高于具体页面、功能和技术实现。

### 3.1 人是目的，AI 是工具

人负责：目标、判断、审美、选择、责任、意义、是否公开、是否继续。

AI 负责：读取、检索、整理、生成、拆解、提醒、执行、辅助沉淀。

AI 不自动发布、不自动精选、不替人做不可逆决策。

### 3.2 点子是自由进入现实的接口

点子不是灵感箱，而是人对现实“不必如此”的回应。一个点子至少要能回答：

- 我对现实哪里不满意
- 我想让它变成什么样
- 谁也受这个问题影响
- 第一步能做什么
- AI 能帮我执行什么
- 人必须判断什么
- 它可能有什么社会价值

### 3.3 内容同时拥有工具价值与存在价值

不是所有内容都必须可复用、可调用、可产品化。

- 工具价值：能解决问题、能复用、能形成方法。
- 存在价值：我活过、感受过、迷茫过、练习过、表达过。

厨艺、书法、日记、吐槽、迷茫不能被系统压扁成“是否有用”。它们可能是 P0 混沌显影，也可能是点子出现的土壤。

### 3.4 界面服务人的理解，结构服务 AI 的执行

```text
界面 = 人类 I/O
结构 / API = AI I/O
内容内核 = 共同事实源
```

人通过页面、卡片、导航、图表、搜索、反馈理解系统。AI 通过 schema、frontmatter、index、graph、API、MCP 理解系统。

### 3.5 数据提供现象，不提供目的

数据告诉我们什么被看见、什么被回应、哪里有误解、哪里有内容缺口。但数据不能决定作者应该成为什么人、应该追求什么、是否应该迎合流量。

数据台必须服务北极星，而不是替代北极星。

### 3.6 公开方向，不公开脆弱

公共数据台可以展示系统正在前进：更新数量、总访问、总点赞、热门内容、最近反馈、路线图、内容缺口。

不公开：具体用户行为、敏感来源、私人日记细节、脆弱表达排名、未准备好公开的问题。

### 3.7 系统帮助人成为自己，不把人系统化

未来平台提供秩序，但不强迫每个人按同一模板生活。入口要有层次：

- 我只是有一个想法
- 我想推进一个目标
- 我想找同行者
- 我想建立自己的系统

### 3.8 重复劳动交给 AI，不可替代的体验留给人

可自动化：整理资料、生成摘要、检索关联、转格式、初步分类、提醒更新、生成备选方案。

不可自动化：为什么要做、是否值得做、什么是美、是否公开、对谁负责、如何表达真实感受、哪个点子值得长期追求。

## 4. 双实践场：iwalk.pro 与 NorthStar

### 4.1 iwalk.pro：个人实践场 / 样板节点

iwalk.pro 是 Walker 的个人前进系统、个人操作系统和样板节点。它回答：

- 一个人如何组织自己的内容、点子、工具、方法、复盘？
- 一个人如何用 AI 降低执行成本？
- 一个人如何保持判断和主体性？
- 一个人的系统如何被人理解、被 AI 读取？

它承载：生活内容、思考文章、AI 学习、工具资源、点子、项目、方法论、个人数据台、Agent 可读接口。

### 4.2 NorthStar：社会实践场 / 多人系统网络

NorthStar / PanGen AI Compass 是多人个人系统网络与点子共创平台。它回答：

- 如何让更多人拥有自己的系统？
- 如何让不同人的点子、能力、内容、需求连接起来？
- 如何让人在平台上交流、协作、共同实现点子？
- 如何让个人价值进入社会价值？

它承载：账号体系、个人系统、点子发布、协作匹配、内容审核、数据平台、权限、通知、公共协作空间。

### 4.3 二者边界

```text
iwalk.pro 是思想源头与样板节点。
NorthStar 是平台化承载与多人网络。

iwalk.pro 不背负多人平台复杂度。
NorthStar 不替代个人表达与方法沉淀。
```

共享：世界观、人决策 AI 执行、点子生命周期、内容结构经验、AI 可读接口经验、数据反馈经验。

不共享：用户体系、平台数据库、权限系统、社区审核、商业化复杂度、敏感合规数据。

## 6. iwalk.pro 信息架构

### 6.1 主导航

主导航采用极简入口：

```text
首页 / 内容 / 关于
```

理由：内容类型未来会增长，不希望主导航被厨艺、书法、AI、日记、项目等绑死。主导航负责稳定入口，分类和视图交给内容页内部筛选。

### 6.2 首页 `/`

首页不是全量导航，而是叙事入口。职责：让人快速理解我是谁、我为什么做、我如何前进、最近有什么值得看、从哪里开始、如何连接。

模块：

- 北极星叙事
- 最近更新
- 正在验证的点子
- 推荐内容路径
- 生活切片
- AI 赋能入口
- 连接入口

### 6.3 内容 `/content`

内容页是“内容宇宙”，不是传统文章列表。

一级视图：

- 全部
- 前进系统
- 生活切片
- 学习记录
- 工具资源
- 作品展示
- 点子生命周期

筛选维度：

| 维度 | 示例 |
| --- | --- |
| form | 文章、笔记、图片、视频、菜谱、书法、项目、工具、教程 |
| domain | AI、编程、厨艺、书法、哲学、生活、情绪、阅读 |
| intent | 记录、教学、复盘、吐槽、验证、展示、连接 |
| status | 想法中、规划中、实践中、修正中、已沉淀、已归档 |
| valueMode | 工具价值、存在价值、二者兼有 |

### 6.4 关于 `/about`

关于页是信任与系统说明页。

Tab：关于我、关于网站、路线图、数据台、技术与 AI、联系方式。

数据台先作为 `/about` 的一个 Tab，不单独占主导航。

### 6.5 旧路由兼容

当前已有 `/posts`、`/tools`、`/ideas`、`/projects`、`/learn` 等概念。迁移时不应立即破坏外链和 SEO。

阶段性策略：

1. 新增 `/content`，保留旧路由。
2. 旧路由逐步成为 `/content` 的视图或重定向目标。
3. 主导航只展示：首页 / 内容 / 关于。
4. 旧路由继续为 SEO、历史链接和外部引用服务。

## 7. 可扩展内容架构

### 7.1 核心原则

内容不按固定栏目写死，而按多个维度组织：

### 7.3 form：内容形态

- article：长文
- note：短笔记
- diary：日记
- rant：吐槽
- gallery：图片集
- video：视频
- recipe：菜谱 / 厨艺
- calligraphy：书法作品
- resource：工具 / 资源
- project：项目
- idea：点子
- lesson：教程

### 7.4 domain：内容领域

- ai
- coding
- product
- philosophy
- life
- cooking
- calligraphy
- reading
- travel
- emotion
- community

### 7.5 intent：内容意图

- think：想清楚一件事
- record：记录生活
- teach：教别人
- share：分享资源
- verify：验证点子
- showcase：展示作品
- reflect：复盘
- connect：寻找同行者
- vent：吐槽 / 释放

### 7.6 valueMode：价值模式

- utility：工具价值
- existence：存在价值
- both：二者兼有

示例：AI 教程偏 utility；项目复盘偏 utility / both；做饭记录偏 existence / both；书法练习偏 existence / both；迷茫日记偏 existence；从迷茫中形成的方法论偏 both。

### 7.7 aiUsePolicy：AI 使用边界

AI 可读不等于 AI 可用。建议分级：

- AI-0：不可读
- AI-1：可读作背景
- AI-2：可引用
- AI-3：可用于推荐
- AI-4：可用于执行任务

示例：

```yaml
aiUsePolicy:
  level: AI-1
  readable: true
  citable: false
  actionable: false
  reason: 情绪记录，仅供理解语境，不作为方法论引用
```

## 9. 人类 I/O 与 AI I/O

### 9.1 人类 I/O

人类用户特点：看图快、耐心短、需要叙事、需要信任、需要情绪和温度、需要明确下一步、不适合一次接收过多结构。

界面职责：把复杂知识系统翻译成人能理解、能行动、愿意继续探索的形式。

界面原则：

- 少主导航，多内部视图
- 默认展示推荐路径，不展示全部复杂度
- 卡片承载内容
- 按钮触发行动
- 图表解释结构
- 搜索帮助找东西
- 数据台解释系统状态
- 反馈入口帮助连接

### 9.2 页面模板

| form | 模板 |
| --- | --- |
| article | 长文阅读页 |
| note | 短卡片 / 时间线 |
| diary | 日记页 / 时间线 |
| gallery | 图片瀑布流 / 相册 |
| video | 视频卡 / 播放页 |
| recipe | 菜谱步骤页 |
| calligraphy | 作品展示页 |
| resource | 工具卡 |
| idea | 点子状态卡 |
| project | 项目案例页 |
| lesson | 教程阶段页 |

### 9.3 AI I/O

AI 不靠视觉理解，而靠稳定 URL、结构化字段、上下文、摘要、关系、引用来源、可调用接口、边界规则。

AI 接口阶段：

1. 静态可读：`/llms.txt`、`/walker-style.md`、frontmatter related。
2. 结构化导出：`/api/index.json`、`/api/graph.json`、JSON-LD、相关文章渲染。
3. MCP：`walker_search`、`walker_get_content`、`walker_get_style`、`walker_get_methodology`、`walker_get_tools`、`walker_graph`、`walker_recommend`。
4. 公开 API + 站内 AI 助手：`/api/v1/content`、`/api/v1/search`、`/api/v1/recommend`、`/ask`。
5. 自动维护：related 建议、过时检测、内容缺口分析、方法论迭代建议、好问题沉淀。

所有自动维护只建议，不自动发布；人确认后生效。

## 10. 反馈数据台

### 10.1 定位

数据台不是炫耀数据，而是反馈与校准层。目标是看见内容有没有被触达、表达是否被回应、哪里有内容缺口、哪些阐述方式需要调整。

### 10.2 数据闭环

```text
内容输出
    ↓
访问 / 点赞 / 评论 / 搜索 / 提问 / 分享
    ↓
数据台
    ↓
AI 总结现象
    ↓
人判断方向
    ↓
内容调整 / 新点子 / 新文章 / 新项目
```

### 10.3 公开数据台

展示：总访问、总点赞、评论数、热门内容、最近被回应内容、本月更新、读者最关心的问题、路线图进度。

不展示：具体用户行为、过细来源、敏感评论、个人脆弱内容排名、私密数据。

### 10.4 私有分析视图

展示：搜索关键词、内容表现矩阵、高访问低互动内容、低访问高互动内容、搜索无结果词、评论问题聚合、AI 内容建议、待更新内容。

### 10.5 内容决策矩阵

| 现象 | 判断 |
| --- | --- |
| 高访问 + 高互动 | 继续展开，做系列 |
| 高访问 + 低互动 | 优化表达，检查是否标题吸引但内容弱 |
| 低访问 + 高互动 | 小众高价值，适合深挖 |
| 低访问 + 低互动 | 归档、重写或降低权重 |
| 高搜索无结果 | 内容缺口，进入选题池 |
| 高评论高提问 | 整理 FAQ / 教程 / 项目 |

## 12. NorthStar 平台架构方向

### 12.1 从个人系统到系统网络

```text
一个人有自己的系统
        ↓
多人拥有自己的系统
        ↓
系统之间通过点子连接
        ↓
平台帮助匹配、协作、执行、沉淀
```

### 12.2 核心对象

```text
UserSystem
├── profile
├── goals
├── ideas
├── capabilities
├── contents
├── tools
├── projects
├── feedback
└── aiPolicy

IdeaCollaboration
├── idea
├── initiator
├── participants
├── roles
├── tasks
├── aiAssists
├── decisions
├── outputs
└── retrospectives
```

### 12.3 平台能力

账号与身份、个人系统主页、点子发布、点子拆解、同行者匹配、协作空间、任务流、AI 辅助、审核与权限、数据台、通知、贡献记录、归属与署名。

### 12.4 必须避免

- 不要先做普通论坛
- 不要先做大而全社区
- 不要让 AI 自动裁决好点子
- 不要用流量排名定义价值
- 不要让平台吞噬个人系统
- 不要让所有人被迫系统化

## 14. 风险与反面清单

| 风险 | 表现 | 防线 |
| --- | --- | --- |
| AI 工具导航站化 | 产品变成工具收藏夹 | 每个工具必须连接真实需要、点子、案例或方法 |
| 效率崇拜 | 只追求更快更多 | 所有自动化必须回答释放出的时间用于什么 |
| 流量内容站化 | 数据驱动变成迎合热点 | 数据只提供现象，不提供目的 |
| 伪社区 | 很多人聊天但没有共同实现点子 | 社区围绕点子、任务、验证、复盘组织 |
| 平台规训 | 个人系统变成持续高效压力 | 允许迷茫、暂停、只记录、不行动 |
| 点子被掠夺 | 公开点子后被拿走 | 时间戳、署名、协作协议、贡献记录、可见范围 |
| AI 责任不清 | AI 输出造成后果 | AI 输出可追溯，人确认后才生效，高风险二次确认 |
| 内容被过度结构化 | 日记、迷茫、生活感受被工具化 | valueMode 区分价值模式，aiUsePolicy 限制 AI 使用 |
| Ferry 教条化 | 把 Ferry 当成不变真理 | 明确 Ferry 是演化协议，受实践反哺 |

## 15. 分阶段路线图

### 阶段 0：设计确认

目标：明确时代定位、Ferry 演化协议、价值宪法、双实践场、内容架构。

产物：架构设计文档、产品架构图、内容模型草案、路线图。

### 阶段 1：iwalk.pro 内容内核重构

目标：主导航收敛为：首页 / 内容 / 关于；建立可扩展 ContentItem 模型。

重点：`/content` 内容宇宙、`/about` 数据台 Tab、旧路由兼容、内容卡片模板、基础筛选。

### 阶段 2：AI 可读接口

目标：让 AI 能读懂 Walker 系统。

重点：`llms.txt`、`walker-style.md`、`index.json`、`graph.json`、`related`、`aiUsePolicy`、JSON-LD。

### 阶段 3：反馈数据台

目标：形成内容反馈闭环。

重点：访问统计、点赞统计、评论聚合、搜索无结果、热门内容、公开数据台、私有分析视图、AI 周报建议。

### 阶段 4：个人系统 Agent 化

目标：让个人 Agent 能调用内容、方法、工具、点子。

重点：MCP、语义搜索、方法论调用、工具推荐、内容过时检查、related 建议。

### 阶段 5：NorthStar 多人系统网络

目标：让更多人拥有自己的系统。

重点：个人系统主页、点子发布、角色与能力、协作匹配、任务拆解、贡献记录、审核、平台数据台。

### 阶段 6：点子共创平台

目标：让人围绕点子共同实现。

重点：协作空间、AI 拆解与辅助执行、项目进度、验证记录、成果沉淀、社会价值评估、跨系统连接。

## 16. 第一阶段 iwalk.pro 推荐落地范围

### 必须做

2. 主导航设计为：首页 / 内容 / 关于。

### 暂不做

- 复杂后台

- 完整数据分析系统

### 原则

先让个人系统成为可理解、可扩展、可读取的样板，再把样板抽象到 NorthStar。

## 17. 关键设计结论

1. Ferry 是演化中的协议，不是固定教条。
2. iwalk.pro 是个人实践场，NorthStar 是社会实践场。
3. 三者通过反馈、阻滞与复盘共同进化。
4. 点子是驱动力，但不是唯一内容类型。
5. 生活内容是点子的土壤，不应被功利化。
6. 主导航保持极简：首页 / 内容 / 关于。
7. 内容页承载多维筛选与多模板展示。
8. 人类界面和 AI 接口读取同一个内容内核。
9. 数据台是反馈校准层，不是流量崇拜层。
10. AI 始终是工具，人始终是目的。
11. 系统要帮助人成为自己，而不是把人系统化。
12. 最终目标是用点子连接人与 AI，也连接人与人，共同减少重复劳动、实现真实改变。
