# Walker 站 · 整体改造方案（单一汇总）

> **本文件是改造相关的唯一方案文档。**  
> **技术栈（已定）：** 全 TypeScript；前端 React + Vite + **pnpm**；后端 **NestJS** + **PostgreSQL**。  
> **产品策略：** 整站重构；**默认体验保真**；仅认定问题区做改造。  
> 含：原则、保真/改造清单、技术栈、观测（功能冷热与失败原因）、模块关系、to-do 与可验证标准。
---

## 1. 一句话

把现网 **iwalk / AdgaiWalker** 用 **TypeScript 全栈**重写：  
前台 **React + Vite**，包管理 **pnpm**；后台 **NestJS + PostgreSQL**。  
大部分前台**看起来、用起来仍是原来的站**；  
同时嵌入「需求可见 → 选题生产 → 检验」推进核；  
不另做一款只含工作台的新产品，不整包复刻旧 Admin 巨石。

---

## 2. 第一性原理（产品宪法）

| 原则 | 含义 |
|------|------|
| 价值 | 问题是否松动 / 困扰是否减少，不是热闹与点赞 |
| 品牌 | 主权在人，可离开平台；平台不办户口本 |
| 卡点 | 不知别人需要什么 → 无法对准生产 → 无法创造价值 |
| AI | 可关；建议 ≠ 验证；不自动替人做主 |
| 标准 | 选题前可自主判断；做成后用结果检验 |
| 一人 | 概念与工程都要可养；页多 ≠ 域多 |

## 4. 保真区 vs 改造区 vs 工程区 vs 延后

### 4.1 A · 体验保真（迁 React，交互与信息架构不动）

| 模块 | 保真标准（可验证） |
|------|-------------------|
| 首页 Bento / 身份 / 导航 / 主题 | 主任务路径可完成；路由心智不变 |
| `/posts` 列表与阅读（TOC/进度等） | 同路径或 301；阅读任务对等 |
| `/learn` `/ideas` `/projects` `/content` | 列表与详情主路径对等 |
| `/about` `/support` | 信息可达 |
| 搜索、点赞、Giscus、RSS/llms 类出口 | 有则行为对等 |
| 已发布 URL 与内容文案 | 不批量改文章；链接不断 |

**验收：** 首页 → 读文 → 搜索 → 换主题，不感到换了产品。

### 4.2 B · 体验改造（迁的同时改）

| 改造点 | 为何 | 改成什么样 | 可验证标准 |
|--------|------|------------|------------|
| `/tools` **问答** | 一头雾水、系统黑话 | 问题导向；场景示例；主强调下一步；弱化记忆条/登录抢戏 | §8 UI Checklist 6/6 |
| **线索进站** | 卡点落在微信/直播 | 管理侧线索池：手动 + 访客提问入库 | intake 后列表可见；body trim≥4 |
| **题苗** | 生产要对准需求 | 点子+选题合并；主选挂 in-pool 线索；两问自检 | promote 无线索→400；primary=1 |
| **执行与检验** | 结果改写标准 | 做→交付→是/否/不清 | §7 可计数闭环 |
| tools **资源列表** | 分类问题后置 | **先保真迁**；分类大改不进首波 | 首波列表 parity |

### 4.3 C · 工程改造（用户可无感）

| 项 | 标准 |
|----|------|
| 全栈语言 | **TypeScript** |
| 前端 | **React + Vite**；**pnpm** |
| 后端 | **NestJS** 模块化架构 |
| 数据库 | **PostgreSQL**（推进核与需持久业务） |
| 写权威 | Nest API 唯一写；前端不伪造业务真相 |
| 内容 | Markdown 为内容真相源（见 §5.3）；阅读体验不变 |
| Admin | 在用页保真迁；推进核新建/收敛；长尾不首迁 |
| WorkItem 全家桶 | **不整包复刻**；闭环语义 → 题苗 + 执行 |

### 4.4 D · 延后 / 不做

点子平台广场、建站 SaaS、Skill/资产晋升全链、ObjectGrant、NorthStar 订单中台、AI 侦察灌库（后置：指令→候选→勾选）、经营模式开关产品化、用工作台替换内容站导航。

## 5. 技术栈计划（已定 + 补齐拍板）

### 5.1 已定

| 层 | 选择 |
|----|------|
| 语言 | **TypeScript**（前后端） |
| 前端 | **React + Vite** |
| 包管理 | **pnpm** |

### 5.2 仓库与包结构（拍板 · 可验证）

| 项 | 标准 |
|----|------|
| 形态 | **本仓库改造成 pnpm monorepo**（保留 git 历史；**不**默认旁路新空仓） |
| 根目录 | 现 `AdgaiWalker-main` 根即 monorepo 根 |
| 应用 | `apps/web` · `apps/admin` · `apps/api` **三个目录必须存在** |
| 共享 | `packages/shared` **必须存在** 且 **零** `react`/`@nestjs/*` 依赖 |
| 包管理器 | 根 `packageManager` 字段含 `pnpm@`；`pnpm-workspace.yaml` 含 `apps/*` `packages/*` |

```text
/   （本仓库根）
├── apps/web/
├── apps/admin/
├── apps/api/              # NestJS
├── packages/shared/
├── content/log/       # MD 真相源路径固定（不迁走，减少摩擦）
├── pnpm-workspace.yaml
└── package.json
```

### 5.3 内容源（拍板 · 可验证）

| 项 | 标准 |
|----|------|
| 文章/指南路径 | **`content/log/**/*.{md,mdx}`** 为唯一内容真相源 |
| 推进核数据 | **仅 PostgreSQL**（线索/题苗/执行/账号/点赞/会话） |
| 前台读内容 | **构建期读 MD**（`packages/shared` 或 `apps/web` 内 gray-matter 扫描）→ 生成静态数据模块；**运行时 web 不直打 Nest 拉全文**（降耦合、利于预渲染） |
| Nest ContentModule | **首版不做**全文 API；若 admin 需预览再后置只读接口 |
| 禁止 | 把历史文章 bulk 导入 PG 当唯一源；禁止 web/admin 各写一套 frontmatter 解析逻辑（**解析函数只在一处**） |
### 5.4 NestJS 模块切分（拍板）

| 模块 | 职责 |
|------|------|
| `AuthModule` | owner/用户会话、cookie |
| `ClueModule` | 原始数据池 |
| `SeedModule` | 题苗 |
| `ExecutionModule` | 交付与检验 |
| `IntakeModule` | 问答入口、nextStep（规则/AI）；**不用** MatchModule 双名 |
| `EngagementModule` | 点赞等保真能力 |
| `AiModule` | AI Port，可关；`AI_ENABLED!==true` 时 Intake 不调模型 |
| 后置 | `ResearchModule`、`PlatformModule`、`ContentModule`（只读 API） |

分层惯例：**Controller → Service → Repository（Prisma）**；业务规则进 Service/`packages/shared`，**禁止** React 内复制状态机。
### 5.5 ORM 与迁移（拍板）

| 项 | 决定 |
|----|------|
| ORM | **Prisma**（迁移清晰、类型友好、一人友好） |
| 迁移 | 所有 schema 变更走 `prisma migrate`，**迁移文件进 git** |
| 生产 | 无 DATABASE_URL 时写接口失败（503），**禁止**静默内存假持久 |

### 5.6 鉴权（拍板）

| 项 | 决定 |
|----|------|
| 阶段主路径 | **httpOnly Cookie 会话**（非前端存 JWT 当唯一凭证） |
| 首波范围 | **登录/注册入口保真存在**；实现可先简化，但路径不能没 |
| Owner | 管理端强校验；推进核写操作须登录（规则按接口） |
| 游客 | 问答允许完整 1 次（§7）；再问须登录 |

### 5.7 Redis 与限流（拍板 · 可验证）

| 项 | 标准 |
|----|------|
| 依赖 | `package.json` **不得** 将 Redis 客户端列为 api **必装** 依赖（首版） |
| 限流实现 | **进程内滑动窗口**（Map+时间戳）；键 = `hash(ip)` 或 userId；窗口 **600s** |
| 限流阈值 | 游客 intake **10/窗口**；已登录 **30/窗口**；超限 HTTP **429** + body.`code=rate-limited` |
| 点赞 | 表 `likes` 在 **PostgreSQL**；`POST` 幂等键可选 ipHash+path |
| 多实例 | 未上 Redis 前部署 **单 api 实例**；README 写明限制 |

### 5.8 搜索（拍板 · 可验证）

| 项 | 标准 |
|----|------|
| 方案 | **Pagefind**；`pnpm` script `build:web` 之后必须跑 `pagefind --site <web-out>` |
| 验收 | 构建产物目录存在 `pagefind/`；⌘K/搜索入口可查询至少 1 篇已有标题 |

### 5.9 前端渲染与 SEO（拍板 · 可验证）

| 项 | 标准 |
|----|------|
| 应用壳 | Vite + React Router SPA |
| 内容页 | **构建期预渲染**：所有 `content/log` 中 `published` 可见的 slug 生成静态 HTML（插件名不限，**结果可验**） |
| 验收 | 对任意已发布 slug：`curl` 静态产物 HTML **含文章 title 字符串**（非空壳 root） |
| 禁止 | 生产发布文章路由仅返回无正文的 CSR 壳 |

### 5.10 部署（拍板 · 可验证）

| 项 | 标准 |
|----|------|
| Web/Admin | 静态托管；**同站反代** `/api` → Nest（cookie 同站） |
| API | 常驻 Node；健康检查路径 **`GET /health` 返回 200** 且 `ok===true` |
| DB | `DATABASE_URL` 必填（生产）；缺失时写接口 **503** `code=storage-unavailable` |
| 云厂商 | **不绑定**；runbook 用「静态托管 + 容器/PaaS + 托管 PG」三件套描述即可 |
| Cookie | `httpOnly`；`SameSite=Lax`；生产 `Secure` |

### 5.11 测试与质量（拍板 · 可验证）

| 项 | 标准 |
|----|------|

### 5.15 会话与 Cookie（拍板 · 可验证）

| 项 | 标准 |
|----|------|
| 会话 Cookie 名 | `walker-session` |
| 载荷 | 至少含 `sid`、`role`（`owner`\|`user`\|`admin`）、`iat`；HMAC 签名密钥 `COOKIE_SECRET`（≥32 字符） |
| 游客 Cookie 名 | `walker-anon`；值 ULID；Max-Age=**2592000**（30d） |
| 配额耗尽响应 | HTTP **401**；JSON `{ "code": "guest-quota-exceeded" }` |
| 限流响应 | HTTP **429**；JSON `{ "code": "rate-limited" }` |
| 存储不可用 | HTTP **503**；JSON `{ "code": "storage-unavailable" }` |
| health | `GET /health` → `{ "ok": true, "db": boolean, "aiEnabled": boolean }`；进程起则 **200**（db 可 false） |

## 6. 领域规则（可验证）

| 规则 | 标准 |
|------|------|
| 线索最短 | `trim(body).length >= 4` |
| 来源 | `manual-self` \| `tools-visitor` \| `wechat` \| `live` \| `other-external` |
| 外部 | 非 `manual-self` |
| 池状态 | `candidate` \| `in-pool` \| `discarded` |
| 游客 1 次 | anon cookie 30d；完整 nextStep 一次后 401 `guest-quota-exceeded` |
| 限流 | 游客 10/10min/IP；owner 30/10min → 429 |
| 主选 | ≥1 条 in-pool 线索；primary 全局=1；旧→backup |
| 两问 | severity + selfInterest ∈ low\|mid\|high |
| 交付 | url 非空 **或** form+note≥4 |
| form | article\|live\|group\|tool\|other |
| 检验 | yes\|no\|unclear；no 须证据≥4 |
| 可计数闭环 | in-pool 线索 + 挂主选 + 合法交付 + outcome∈{yes,no} +（no⇒证据） |
| 毕业 | 可计数≥5 且 yes≥2 且 外部闭环≥1；或 S1 上线后 56 天 + 复盘文档 |
| 空转 | 连续 3 次 no/unclear → spin-risk 警告，不锁写 |
| ID/时间 | ULID；ISO-8601 UTC |
| 分页 | 默认 50，max 100 |

## 13. 核心模块与关系（依赖倒置）

> 标注约定：  
> - **依赖** A → B：编译/设计期需要 B 的抽象（优先接口）  
> - **调用** A → B：运行期发起同步请求/函数调用  
> - **触发** A → B：事件/副作用，B 可失败而不推翻 A 主结果（若无特别说明）  
> - **实现** A ⇒ I：A 是接口 I 的具体适配器  
> 箭头方向 = **谁指向谁**（依赖倒置：高层依赖接口，低层实现接口）。

### 13.1 核心模块（一句话职责）

#### 前端壳（易变）

| 模块 | 一句话职责 |
|------|------------|
| **WebApp** | 公开站：保真浏览/阅读 + 问答入口展示与提交 |
| **AdminApp** | 管理端：线索/题苗/执行操作 + 在用后台页 |
| **ContentReadModel** | 构建期从 MD 生成只读内容视图（列表/详情数据），供 Web 渲染 |

#### 契约与领域（稳定核 · 无框架）

| 模块 | 一句话职责 |
|------|------------|
| **SharedKernel** | 枚举、DTO、错误码约定、闭环/主选等**纯规则函数**（无 I/O） |
| **DomainPorts** | 对外端口接口：`ClueRepository`、`SeedRepository`、`ExecutionRepository`、`AiPort`、`Clock`、`IdGenerator` 等 |

#### Nest 应用服务（用例编排）

| 模块 | 一句话职责 |
|------|------------|
| **IntakeApplication** | 接待卡点：校验 → 入库线索 → 生成 nextStep（规则或 AI） |
| **ClueApplication** | 线索池用例：手动建、冷补、丢弃、列表查询 |
| **SeedApplication** | 题苗用例：存款/挂线索/主选/两问/唯一 primary |
| **ExecutionApplication** | 执行用例：deliver、review、推进状态 |
| **MetricsApplication** | 聚合可计数闭环、毕业进度、spin-risk |
| **AuthApplication** | 登录会话、角色、游客 anon 识别与配额判定 |
| **EngagementApplication** | 点赞等保真互动（行为 parity） |
| **AiApplication** | 对 `AiPort` 的应用层封装（可关、超时、打标 aiUsed） |

#### 适配器 / 基础设施（可替换实现）

| 模块 | 一句话职责 |
|------|------------|
| **HttpApi（Nest Controllers）** | HTTP 进出、DTO 校验、鉴权守卫挂载，不写领域规则 |
| **PrismaClueRepo 等** | 用 Prisma/PG **实现** Repository 端口 |
| **RuleNextStepAdapter** | 无模型时按关键词桶生成 nextStep（**实现** NextStep 策略接口） |
| **LlmNextStepAdapter** | 有 key 且 AI_ENABLED 时调模型（**实现** 同一策略接口） |
| **SessionCookieAdapter** | 签发/校验 `walker-session` / `walker-anon` |
| **InMemoryRateLimiter** | 进程内滑动窗口限流（首版实现 RateLimitPort） |
| **ConfigAdapter** | 读取 env（DATABASE_URL、AI_ENABLED、COOKIE_SECRET…） |
| **ObservabilityApplication** | 写入功能事件、聚合冷热/漏斗/失败码（§14） |
| **FeatureEventRepository（端口）** | 事件持久化抽象 |
| **PrismaFeatureEventRepo** | PG **实现**事件表 |

#### 后置（首版不实现，占位）

| 模块 | 一句话职责 |
|------|------------|
| **ResearchApplication** | 人下令侦察 → 候选线索（不自动 in-pool） |
| **PlatformApplication** | 上架/listed 可见性 |

### 13.2 模块间关系（方向 + 理由）

#### A. 壳 → 核（只经 HTTP，不直触库）

| 关系 | 类型 | 理由 |
|------|------|------|
| WebApp → HttpApi | **调用** | 浏览器只打 REST；不 import Nest/Prisma |
| AdminApp → HttpApi | **调用** | 同上；写操作带会话 Cookie |
| WebApp → ContentReadModel | **依赖** | 构建/运行读本地生成的内容数据，不经 API 拉全文（首版） |
| WebApp / AdminApp → SharedKernel | **依赖**（可选） | 仅共享类型/常量；**禁止**在壳内复制状态机实现 |

#### B. 接口层 → 应用层

| 关系 | 类型 | 理由 |
|------|------|------|
| HttpApi → AuthApplication | **调用** | Guard/中间件先鉴权再进业务 |
| HttpApi → IntakeApplication | **调用** | `POST .../intake` |
| HttpApi → ClueApplication | **调用** | 线索 CRUD/列表 |
| HttpApi → SeedApplication | **调用** | 题苗/主选 |
| HttpApi → ExecutionApplication | **调用** | 交付/检验 |
| HttpApi → MetricsApplication | **调用** | 进度查询 |
| HttpApi → EngagementApplication | **调用** | 点赞等 |
| HttpApi → SharedKernel | **依赖** | 错误码、入参枚举；Controller 不写规则分支 |

#### C. 应用层 → 端口（依赖倒置 · 核心）

| 关系 | 类型 | 理由 |
|------|------|------|
| IntakeApplication → ClueRepository（端口） | **依赖** | 入库不绑 Prisma |
| IntakeApplication → NextStepStrategy（端口） | **依赖** | 可关 AI：换适配器即可 |
| IntakeApplication → AuthApplication / GuestQuotaPort | **调用/依赖** | 判游客是否耗尽 |
| IntakeApplication → RateLimitPort | **依赖** | 限流实现可换 |
| IntakeApplication → SharedKernel | **依赖** | body 最短、纯校验 |
| ClueApplication → ClueRepository | **依赖** | 同上 |
| SeedApplication → SeedRepository + ClueRepository | **依赖** | 主选须查线索 in-pool |
| SeedApplication → SharedKernel | **依赖** | `assertPrimaryHasClue`、两问形状 |
| SeedApplication → ExecutionApplication | **调用**（或域事件） | promote 时创建 doing 执行卡；**不**反向依赖 |
| ExecutionApplication → ExecutionRepository + SeedRepository | **依赖** | 校验主选与交付 |
| ExecutionApplication → SharedKernel | **依赖** | `isCountableLoop` 等 |
| MetricsApplication → 各 Repository（只读端口） | **依赖** | 聚合查询；不写业务迁移 |
| MetricsApplication → SharedKernel | **依赖** | `graduationProgress` |
| AiApplication → AiPort | **依赖** | 模型厂商可换 |
| AuthApplication → SessionPort + AccountRepository | **依赖** | 会话存储可换 |
| EngagementApplication → LikeRepository | **依赖** | 点赞落 PG |

**禁止的依赖（对抗）：**

- Repository 实现 → Application（反向）  
- Prisma* → SharedKernel 内写业务 if 森林（规则应在 shared 纯函数）  
- Application → 具体 `OpenAISdk` 类（必须经 AiPort）  
- WebApp → Prisma / Repository  

#### D. 实现关系（适配器 ⇒ 端口）

| 关系 | 类型 | 理由 |
|------|------|------|
| PrismaClueRepo ⇒ ClueRepository | **实现** | PG 适配 |
| PrismaSeedRepo ⇒ SeedRepository | **实现** | 同上 |
| PrismaExecutionRepo ⇒ ExecutionRepository | **实现** | 同上 |
| RuleNextStepAdapter ⇒ NextStepStrategy | **实现** | AI 关闭默认 |
| LlmNextStepAdapter ⇒ NextStepStrategy | **实现** | AI 开启 |
| SessionCookieAdapter ⇒ SessionPort | **实现** | Cookie 会话 |
| InMemoryRateLimiter ⇒ RateLimitPort | **实现** | 首版无 Redis |
| （后）RedisRateLimiter ⇒ RateLimitPort | **实现** | 多实例时替换，应用层不改 |

#### E. 触发（弱耦合副作用）

| 关系 | 类型 | 理由 |
|------|------|------|
| IntakeApplication 成功后 → （可选）Metrics 缓存失效 | **触发** | 非必须；首版可同步读库 |
| ExecutionApplication review 后 → Metrics 重算 | **触发** | 读模型更新；失败不回滚 review |
| AiApplication 失败 → Intake 仍返回规则 nextStep | **触发/降级** | 主路径诚实降级，不装 AI 成功 |

### 13.3 依赖方向总览（允许的箭头）

```text
WebApp / AdminApp
    │ 调用 HTTP
    ▼
HttpApi (Controllers/Guards)
    │ 调用
    ▼
*Application（用例）
    │ 依赖接口
    ▼
DomainPorts  +  SharedKernel（纯规则）
    ▲ 实现
    │
Adapters: Prisma* / RuleNextStep / Llm* / Cookie / RateLimiter
    │
    ▼
PostgreSQL / 外部 LLM
```

### 13.4 和「别写死 / 逻辑分离」的对应

| 经验 | 在模块图上的体现 |
|------|------------------|
| 别写死 | 限流/AI/会话都是 **Port + 可替换实现**；配置进 ConfigAdapter |
| UI↔业务 | 壳只到 HttpApi |
| 接口↔领域 | Controller 薄，Application 编排 |
| 领域↔存储 | Repository 端口 |
| 稳核↔变壳 | SharedKernel/Ports 稳；Web 样式与布局可变 |
| 读写分离 | ContentReadModel（MD 构建读）≠ Clue 写模型（PG） |

## 14. 站内观测（功能冷热 + 失败原因）

> **目标：** 监测**本站/本应用内**功能使用与结果，而非外部爬数。  
> **要回答：**  
> 1）哪些功能常用、哪些不常用？  
> 2）「没用/没成」是**没有这个功能**，还是**别的因素**（找不到、要登录、报错、太慢等）？  
> **对抗裁定摘要：** 做「功能漏斗 + fail_code + 业务结果」；不做热力图/录屏/默认第三方分析/站外自动采/广告画像。

### 14.1 三种数据（禁止混用结论）

| 类型 | 作用 | 存哪 |
|------|------|------|
| **功能事件** | 曝光/尝试/成功/失败原因 | PostgreSQL 事件表 |
| **业务结果** | 是否有用（检验、内容反馈） | 推进核表 + 反馈表 |
| **运维日志** | 系统是否坏了 | 日志通道（短留存）；**不**当「功能冷热」主依据 |

### 14.2 功能单元 feature_key

每个可监测能力有稳定 id（英文点分）。**首版字典最低集合：**

| feature_key | 成功定义（可验证） |
|-------------|-------------------|
| `nav.home` | 首页主壳渲染完成（expose 即可计到达） |
| `post.list` / `post.read` | 进入列表/进入文章详情 |
| `search.query` | 提交搜索；props.had_results=true/false |
| `match.intake` | intake API 业务成功并返回 nextStep |
| `tools.resource_click` | 点击资源链接（outbound 可记） |
| `like.click` | 点赞 API 成功 |
| `auth.login` | 登录成功 |
| `clue.create_manual` | admin 手动建线索成功 |
| `seed.promote` | 主选成功 |
| `execution.review` | 检验写入成功（outcome 进 props） |
| `content.feedback` | 内容反馈提交成功 |

新增功能必须：**先登记 feature_key，再上线埋点**（否则 M 验收不算观测完成）。

### 14.3 标准漏斗与事件名

每个 feature 遵循：

```text
expose → attempt → success
              ↘ fail（必带 fail_code）
```

| 事件名 | 何时打 | 谁打（优先） |
|--------|--------|--------------|
| `feature_expose` | 进入功能页/区 | 前端（path→key 映射） |
| `feature_attempt` | 主 CTA/提交开始 | 前端或 API 入口 |
| `feature_success` | 业务成功 | **服务端优先** |
| `feature_fail` | 业务失败 | **服务端优先** |

**单条事件最小字段（可验证）：**

- `id`（ULID）、`at`（ISO UTC）、`feature_key`、`event`（expose\|attempt\|success\|fail）  
- `actor_type`：`guest`\|`user`\|`owner`  
- `fail_code`：仅 fail 时必填  
- `entity_type` / `entity_id`：可选（post slug、clue id…）  
- `props`：小 JSON（如 had_results、outcome）；**禁止**存密码、长 PII  

### 14.4 fail_code 枚举（区分「没功能」vs「别的因素」）

| fail_code | 含义 | 解读倾向 |
|-----------|------|----------|
| `auth_required` | 要登录 | 有功能，门槛 |
| `quota_exceeded` | 游客次数用完 | 有功能，配额 |
| `rate_limited` | 限流 | 有功能，防刷 |
| `validation_error` | 入参不合法 | 有功能，用法/校验 |
| `server_error` | 5xx/未处理异常 | 有功能，坏了 |
| `not_found` | 404 | 路径/内容不存在 |
| `empty_result` | 搜索等无结果 | **缺口信号**（内容或能力词） |
| `feature_missing` | 产品明确未实现 | **真缺功能** |
| `ai_failed_degraded` | AI 失败已降级 | 有功能，质量/依赖（仍可能 success） |
| `user_abandon` | 前端启发式放弃（可选后置） | 体验中断 |

**判断规则（可验证、给人看的）：**

| 模式 | 结论 |
|------|------|
| 无对应 success，且 empty_result / 线索原文反复要某能力 | **缺功能或缺内容** |
| 尝试高 + auth/quota/rate/validation 为主 | **有功能，摩擦** |
| 尝试高 + server_error 为主 | **有功能，故障** |
| expose≈0 且入口应存在 | **有功能，难发现**（先不删） |
| success 稳定但量极低 | **价值/场景弱** |

**禁止：** 用点赞数或 PV 单独断言「有用/没用」。

### 14.5 看板（admin，可验证）

最低一页，周维度默认：

1. **功能冷热：** 各 feature_key 的 expose / attempt / success 计数  
2. **失败 Top：** fail_code 占比 Top 5（全局 + 可点进某 feature）  
3. **有用（结果）：** 检验 yes 数、内容反馈 useful 数（与事件分列，**不合并分母**）  
4. **入口健康：** intake 成功率、quota/rate/server_error 次数  

**验收：** 连续 7 天有数据时，能指出 1 个最常用 + 1 个最冷 + 1 个主失败码。

### 14.6 运维日志（底线）

| 级别 | 记 | 不记 |
|------|----|------|
| error | 异常、DB 失败、未捕获 | 密码、cookie 全文 |
| warn | 限流、配额、AI 降级 | 每个 200 |
| info | 启动、migrate | 刷屏 debug |

- 每请求 **requestId**（响应头可回传）  
- 运维日志保留建议 **7–14 天**；产品事件在 PG，保留策略可更长（业务需要）  
- 观测故障**不得**拖垮主请求（写入失败打 warn，主流程仍按业务成功/失败返回）

### 14.7 对抗裁定：纳入 / 延后 / 不做

| 状态 | 能力 |
|------|------|
| **纳入 P0** | feature 字典；服务端 success/fail+code；推进核结果；health；error 日志 |
| **纳入 P1** | admin 冷热+失败码看板；漏斗聚合；搜索无结果 |
| **纳入 P2** | page_view；内容反馈；点赞仅辅列 |
| **延后** | 阅读深度 50%/90%；性能 Web Vitals；user_abandon；多实例再上 Redis 指标 |
| **不做** | 热力图/录屏；默认 GA/PostHog 等第三方（除非书面改方案）；站外平台自动爬取；广告式跨站画像 |

### 14.8 隐私（与宪法一致）

- 站内观测，目的为改进产品与选题，不卖数据  
- 少 PII；线索正文仅业务必要；搜索词优先 had_results，关键词可 hash/截断  
- 不采长期明文 IP 作身份（限流可用短时 hash）  
- 不默认采集站外账号体系数据  

### 14.9 与里程碑挂接

| 里程碑 | 观测交付 |
|--------|----------|
| M1 | health + 结构化 error 日志 + requestId |
| M3 | `match.intake` 全漏斗；fail_code；feature 字典初版 |
| M4 | promote/review 事件；有用结果进 metrics |
| M5 | page_view、搜索无结果、内容反馈；**冷热榜+失败码 Top 页** |

### 14.10 模块关系（观测）

| 关系 | 类型 | 理由 |
|------|------|------|
| \*Application 成功/失败 → ObservabilityApplication | **触发** | 业务主路径优先；埋点失败不改写业务结果 |
| ObservabilityApplication → FeatureEventRepository | **依赖** | 端口，可换存储 |
| HttpApi → ObservabilityApplication | **调用** | 查询看板、可选前端 expose 上报 |
| AdminApp → HttpApi（metrics/features） | **调用** | 只读观察 |
| 禁止 WebApp 直写库记事件 | — | 须经 API，便于鉴权与校验 |
