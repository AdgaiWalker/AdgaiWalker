# Phase 1 Plan: Invited User -> Need Case -> Admin Review

Generated: 2026-06-14

Stage: AdgaiWalker 个人系统 / 发布接口桥接

## 1. 目标

本阶段只验证一条纵向切片：

```text
访客输入邀请码
-> 成为邀请用户
-> 填写或跳过轻量画像
-> 提交一个真实需求
-> Agent 读取用户上下文并给出推荐
-> 系统保存 Need Case
-> 用户提交反馈
-> 管理员在后台复盘 Need Case
```

这个阶段不追求完整后台、完整内容创作、完整 Agent 平台或完整社区系统。它只回答一个问题：

```text
真实邀请用户的需求，能不能被 Agent 处理，并变成 Walker 后台可复盘、可创作的材料？
```

## 2. 范围

### 2.1 In Scope

- 邀请码公开验证入口。
- invited 轻量会话。
- public / invited / admin 状态判断。
- 用户画像 GET / POST。
- `/api/match` 读取用户上下文。
- Need Case 数据结构和存储接口。
- Agent Recommendation 保存到 Need Case。
- User Feedback 保存并回写 Need Case。
- Admin Review 最薄列表。
- Safety Layer 第一批失败处理。
- 最小测试覆盖。

### 2.2 Out of Scope

- 完整注册登录。
- 邮箱/手机号绑定。
- 找回密码。
- 用户主页。
- 关注、粉丝、积分、等级。
- `graph.json`。
- Publish Interface。
- Skill Admission。
- 内容草稿自动生成。
- 多 Agent 角色 UI。
- NorthStar 社区网络。

## 4.1 Need Case 迁移规则

本阶段已确认：

```text
Need Case 直接取代 DemandEvent 的主业务地位。
不写兼容代码。
不做双写。
不做长期并行迁移。
不为旧调用方保留 shim。
```

如果旧代码依赖 `DemandEvent`：

```text
直接改调用方
直接改 store / service / admin 读取源
直接改测试
```

不要做：

```text
NeedCaseAdapter
DemandEventAdapter
同时写入 Need Case 和 DemandEvent
保留两套后台主数据源
```

## 5. 服务边界

| Service | 职责 | 依赖 Port |
| --- | --- | --- |
| InviteAccessService | 验证邀请码，建立 invited 会话。 | InviteCodeRepositoryPort, InvitedSessionRepositoryPort, SafetyPolicyPort |
| UserProfileService | 读取、保存、更新当前邀请用户画像。 | UserProfileRepositoryPort, InvitedSessionRepositoryPort |
| UserContextService | 汇总 authState、session、profile、current need。 | InvitedSessionRepositoryPort, UserProfileRepositoryPort |
| AgentOrchestrator | 编排需求处理：读上下文、匹配、保存 Need Case、返回推荐。 | UserContextPort, MatchingPort, NeedCaseRepositoryPort, SafetyPolicyPort |
| FeedbackService | 保存反馈并回写 Need Case。 | FeedbackRepositoryPort, NeedCaseRepositoryPort |
| AdminReviewService | 后台复盘 Need Case 和处理状态。 | NeedCaseRepositoryPort, FeedbackRepositoryPort, AdminReviewRepositoryPort |
| SafetyService | 处理失败、降级、拒绝、记录、交给人。 | SafetyPolicyPort, IncidentRepositoryPort |

## 6. API 切片

| API | 方法 | 第一版行为 |
| --- | --- | --- |
| `/api/invite/verify` | POST | 验证邀请码；成功后建立 invited session；失败不创建 session。 |
| `/api/profile` | GET | invited/admin 读取当前会话画像。 |
| `/api/profile` | POST | invited/admin 创建或更新当前会话画像。 |
| `/api/match` | POST | 读取 User Context；执行匹配；保存 Need Case；返回 recommendation。 |
| `/api/match-feedback` | POST | 保存反馈；回写 Need Case。 |
| `/api/admin/review` | GET | admin 查看 Need Case 列表。 |
| `/api/admin/review` | POST | admin 更新 reviewStatus / note / topicCandidateId。 |

## 7. 安全层第一批规则

| 场景 | 处理 |
| --- | --- |
| 邀请码错误 | 拒绝准入，不创建用户会话。 |
| 邀请码过期或用完 | 拒绝准入，返回可理解原因。 |
| 画像保存失败 | 允许继续使用，但 User Context 标记低置信度。 |
| AI 调用失败 | 使用本地规则匹配降级。 |
| Need Case 保存失败 | 不影响用户响应，但记录 Incident。 |
| 反馈保存失败 | 返回失败提示，允许用户重试。 |
| 权限不确定 | 默认拒绝，不默认放行。 |
| 敏感信息出现 | 脱敏后再保存，必要时不保存原始需求。 |
| Agent 低置信度 | 不强行推荐，先追问。 |
| 连续失败 | 进入管理员待处理队列。 |

## 10. Stop Rule

如果 5-10 个邀请用户里没有人提交可复盘的真实需求，或者管理员后台无法从 Need Case 中判断出任何明确选题，则暂停扩展功能，回到：

```text
需求入口文案
Agent 匹配质量
画像问题是否过多
Need Case 字段是否抓错重点
```

不要继续扩 `graph.json`、Publish Interface、Skill Admission 或复杂后台。

如果执行中发现只有依靠兼容层、双写或长期双轨才能继续推进，也要暂停，回到模型边界本身；不要为了“先跑起来”把错误模型固化。
