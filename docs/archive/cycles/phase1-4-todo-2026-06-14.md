# To-do List: Phase 1 Invited User -> Need Case -> Admin Review

## Execution Decision
- **Judgment**: Go
- **Reason**: 路线已经足够清楚，最大模型决策已经确认：`Need Case` 直接取代 `DemandEvent` 主业务地位，而且不写兼容代码。

## Completion Definition
一个被邀请并通过邀请码的用户，能够进入 iwalk.pro、提交真实需求、获得 Agent 推荐；系统能保存这条 Need Case 和用户反馈；管理员能在后台复盘这条 Need Case，并决定是否值得进入后续选题或内容创作。

## Todo Boundary
- **Type**: product
- **Boundary**: 只覆盖邀请制准入、轻量画像、Need Case、Agent Recommendation、User Feedback、Admin Review 这一条纵向切片。
- **Non-goals**:
  - 完整账号系统
  - graph.json
  - Publish Interface
  - Skill Admission
  - 内容草稿自动生成
  - 多 Agent 角色产品化
- **Deferred work**:
  - TopicCandidate 自动生成和发布回填
  - 经验验证和方法成熟度评估
  - 内容创作闭环和作品交付
  - AI 接口 v2
- **Verification rule**: 必须同时证明“用户侧链路可用”和“后台复盘链路可用”，仅完成 API 或页面不算达标。
- **Evidence source**: API 行为、存储记录、后台页面、测试、手动试用轨迹
- **Pass criteria**: 一条真实 invited user 链路可以从邀请码验证走到后台复盘；至少保存一条带画像快照、推荐结果和反馈的 Need Case；非 admin 无法访问后台复盘数据。
- **Confidence note**: 这是强代理指标而不是最终产品成功指标，但它足够证明第一条主链路成立。
- **Judgment owner**: Walker + 行为验证结果

## Drift Diagnosis
- **Todo drift**: 旧 plan 里有部分内容已经滑向 TopicCandidate、内容发布和 Skill 候选，这些不直接证明第一轮完成定义。
- **Phase drift**: 旧 plan 更像模块清单，还没有把“先证明 invited user 到 admin review”放在最前面。
- **Validation drift**: 旧 plan 偏向“新增 API/Service/Port”，而不是“真实一条链能跑通”。
- **Compatibility drift**: 风险不再是“关系没定清”，而是执行时滑回双轨、双写或 shim。
- **Cleanup drift**: 后台增强、图谱、内容创作相关动作容易混进第一轮，削弱证明点。

## Priority Rationale
- 邀请码准入和会话先做，因为它决定谁是用户。
- User Context 早于匹配改造，因为 `/api/match` 是否读取画像是本阶段的核心证明点。
- Need Case 要比后台 UI 先定，因为后台只是读取 Need Case。
- Admin Review 可以很薄，因为它只负责证明“后台能复盘”，不负责第一轮完成内容创作。

## Phases

### Phase 1: Auth State And Invited Session
- **Purpose**: 先把“谁是用户”钉住。
- **Entry condition**: 当前已有邀请码 store 和临时 admin invite verify。
- **Phase rules**:
  - 只做邀请制，不做开放注册。
  - `public` 不是用户。
  - invited 状态只能由邀请码验证产生。
  - 权限不确定时默认拒绝。
  - 不为旧认证入口保留长期兼容层。
- **Todos**:
  - [ ] 新增 `/api/invite/verify`
    - **Surface**: API
    - **Proof**: 有效邀请码返回 invited session；无效邀请码不创建 session
    - **Depends on**: none
  - [ ] 新增 `InvitedSessionRepositoryPort`
    - **Surface**: Port / store
    - **Proof**: 能创建、读取、失效 session
    - **Depends on**: none
  - [ ] 实现 InviteAccessService
    - **Surface**: service
    - **Proof**: API route 只调用 service，不内嵌业务逻辑
    - **Depends on**: `InviteCodeRepositoryPort`, `InvitedSessionRepositoryPort`
- **Exit proof**: 访客可通过有效邀请码获得 invited 状态；无效邀请码被拒绝。
- **Stop condition**: 如果 invited 状态需要引入完整账号或复杂权限，立即暂停并回到边界定义。

### Phase 2: Profile And User Context
- **Purpose**: 建立 Agent 可用的最小用户上下文。
- **Entry condition**: invited session 已可创建和读取。
- **Phase rules**:
  - 核心画像问题不超过 5 个。
  - contact 默认可选，若保留必须脱敏。
  - 画像保存失败不阻断用户继续提问。
  - 不为旧画像读取方式保留双路实现。
- **Todos**:
  - [ ] 实现 `/api/profile` GET / POST
    - **Surface**: API
    - **Proof**: invited 可读写，public 不可访问
    - **Depends on**: invited session
  - [ ] 实现 UserProfileService
    - **Surface**: service
    - **Proof**: 画像逻辑从 route 中分离
    - **Depends on**: `UserProfileRepositoryPort`
  - [ ] 实现 UserContextService
    - **Surface**: service
    - **Proof**: 能返回 authState + profile + current question context
    - **Depends on**: invited session, profile
- **Exit proof**: invited 用户能保存画像，`/api/match` 可读取上下文依赖。
- **Stop condition**: 如果画像字段开始扩张为 CRM 或社区账号资料，立即收缩。

### Phase 3: Need Case And Agent Recommendation
- **Purpose**: 让用户需求真正落进系统，而不是停留在聊天响应。
- **Entry condition**: User Context 已可读取。
- **Phase rules**:
  - `/api/match` 只做 HTTP 层，调用 `AgentOrchestrator.handleNeed()`。
  - AI 失败时必须 fallback，不允许整条链中断。
  - Need Case 保存失败不影响用户响应，但必须记录 incident。
  - Need Case 直接取代 DemandEvent 主地位，不写兼容代码。
- **Todos**:
  - [ ] 定义 Need Case schema，并列出从 `DemandEvent` 吸收的字段
    - **Surface**: doc / schema
    - **Proof**: schema 明确，且不保留双轨
    - **Depends on**: none
  - [ ] 实现 `NeedCaseRepositoryPort`
    - **Surface**: Port / store
    - **Proof**: 可保存 profile snapshot、summary、recommendation、safety flags
    - **Depends on**: Need Case schema
  - [ ] 实现 `AgentOrchestrator`
    - **Surface**: service
    - **Proof**: `/api/match` 收敛为入口层
    - **Depends on**: `UserContextPort`, `MatchingPort`, `NeedCaseRepositoryPort`, `SafetyPolicyPort`
  - [ ] 改造 `/api/match`
    - **Surface**: API
    - **Proof**: 可以返回 recommendation 并保存 Need Case
    - **Depends on**: `AgentOrchestrator`
- **Exit proof**: 至少一条 invited user 请求能生成带画像快照和 recommendation 的 Need Case。
- **Stop condition**: 如果实现开始依赖双写、shim 或长期双轨，暂停扩展后台和反馈，先收缩回单一主模型。

### Phase 4: Feedback And Admin Review
- **Purpose**: 证明后台复盘链路成立。
- **Entry condition**: Need Case 已能保存。
- **Phase rules**:
  - 先做最薄后台列表，不做复杂图表。
  - feedback 必须回写到 Need Case。
  - 非 admin 一律不可访问 review 数据。
- **Todos**:
  - [ ] 改造 `/api/match-feedback`
    - **Surface**: API
    - **Proof**: feedback 可关联到 Need Case
    - **Depends on**: Need Case
  - [ ] 实现 FeedbackService
    - **Surface**: service
    - **Proof**: feedback 保存逻辑与 route 分离
    - **Depends on**: `FeedbackRepositoryPort`, `NeedCaseRepositoryPort`
  - [ ] 新增 `/api/admin/review`
    - **Surface**: API
    - **Proof**: admin 可查看 Need Case 列表并更新 reviewStatus
    - **Depends on**: Need Case, Feedback
  - [ ] 新增最薄 Admin Review UI
    - **Surface**: admin UI
    - **Proof**: 后台能看到画像摘要、需求摘要、recommendation、feedback、reviewStatus
    - **Depends on**: `/api/admin/review`
- **Exit proof**: 管理员能从后台看到至少一条完整 Need Case，并可做 `accepted / deferred / ignored` 标记。
- **Stop condition**: 如果后台页面开始演化成大而全工作台，先停，保持只服务第一条复盘链。

## Dry-Run Findings
- `Need Case` 已确认为单一主模型，后续实现要主动避免滑回 `DemandEvent` 双轨。
- Admin Review 不需要单独产品化，只要能被管理员读取和更新状态就够。
- `TopicCandidate` 在本阶段只需要“能转过去”的接口，不需要自动生成。
- 如果先做后台 UI 再做 Need Case，实施会倒序，容易变成看空数据的页面。
- 执行中必须主动抵制“先保留兼容，后面再收”的诱惑；那会把第一轮重新拖回旧模型。

## Final Validation
- 手动用 1 个有效邀请码走完整链路。
- 验证 1 个无效邀请码被拒绝。
- 验证 1 条 Need Case 保存了 profile snapshot、summary、recommendation、feedback。
- 验证非 admin 无法访问 review 数据。
- 验证管理员能将 1 条 Need Case 标记为 `accepted / deferred / ignored`。

## First Execution Step
先定义 Need Case schema，并列出当前 `DemandEvent` 中哪些字段被吸收、哪些字段被删除。然后直接从 `/api/match` 和后台读取链开始改到单一主模型。
