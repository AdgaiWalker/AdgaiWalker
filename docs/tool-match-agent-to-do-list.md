# 需求索引匹配 Agent 安全与选题库 To-do List

日期：2026-06-06

## 0. 目标

在 `/tools` 的需求索引匹配功能基础上，建立一个安全、克制、可沉淀选题的站内资料推荐 Agent。

核心目标：

- 帮用户把“需求”匹配到站内已有工具、文章、资源、社群和学习路径。
- 不做外部搜索，不重新加工整篇文章，只生成一句低认知成本的串联语。
- 不收集个人敏感数据，只收集匿名、脱敏后的需求。
- 通过定时集中处理，把真实需求沉淀为自媒体选题库。

## 1. 产品原则

### 1.1 Agent 角色

Agent 不是“问答机器人”，而是“站内资料路由器”。

它只做三件事：

1. 听懂用户现在想解决什么问题。
2. 从站内资料索引中选出 3-5 条最有帮助的资料。
3. 用一句话说明这些资料如何串起来解决问题。

### 1.2 内容原则

- 不外部搜索。
- 不编造工具。
- 不重写文章。
- 不把你的内容重新加工成长篇答案。
- 推荐文章时加入理解兜底：看不懂时，把原文丢给 AI，让它用生活里的例子讲一遍。

### 1.3 数据原则

- 不收集身份，只收集需求。
- 不保存完整对话，默认保存脱敏后的需求样本。
- IP 只用于限流，不进入选题库。
- 未成年人或青少年场景进入保护模式。
- 用户可以关闭“匿名加入选题库”。

## 2. 优先级总览

### P0：上线前必须完成

- 修复前端消息渲染 XSS 风险。
- 增加后端请求长度、轮数、频率和成本限制。
- 增加隐私提示和匿名选题开关。
- 禁止保存个人敏感数据。
- 禁止模型输出未在站内索引中的资料。
- 清理服务端日志，避免记录完整 prompt、用户原文和上游错误正文。

### P1：核心体验闭环

- 把自由聊天回复改成“资料包结果”。
- 建立站内资料索引。
- 让模型只选择资料 ID 和写一句串联语。
- 记录会话开始时间、结束时间、消息轮数和推荐资料。
- 加入“清空对话”和“本次不加入选题库”。

### P2：选题库自动化

- 建立匿名需求事件表。
- 建立每日定时批处理。
- 聚类相似需求。
- 生成选题候选卡。
- 人工确认后进入正式选题库。

### P3：数据洞察与内容反馈

- 统计高频需求、工具误区、内容缺口。
- 记录用户点击了哪些推荐资料。
- 给每个选题关联站内资料和后续发布内容。
- 做一个私有的选题库/需求雷达页面。

## 3. 行为衔接逻辑

### 3.1 用户路径

```text
用户进入 /tools
→ 看到“别急着用 Codex，先说你要做什么”
→ 阅读隐私提示
→ 可选选择人群/阶段
→ 输入需求
→ Agent 返回资料包
→ 用户点击站内资料
→ 会话结束或超时
→ 保存匿名需求事件
→ 每日批处理进入选题候选
```

### 3.2 会话结束规则

一个会话在以下任一条件满足时结束：

- 用户点击关闭。
- 用户点击清空对话。
- 用户 10 分钟无新消息。
- 页面关闭或刷新，记录最后活跃时间。

结束时间计算：

```text
ended_at = explicit_close_at || clear_at || last_active_at + 10min
```

### 3.3 数据进入选题库逻辑

```text
完整聊天
→ 不长期保存

脱敏需求
→ 进入 pending_demand_events

每日批处理
→ 需求摘要、分类、聚类、内容缺口判断

人工确认
→ 进入 topic_candidates / topic_library
```

## 4. 数据分层设计

### 4.1 会话层：match_sessions

只记录必要的会话元数据。

字段建议：

- `session_id`：随机 ID，不是用户身份。
- `started_at`：会话开始时间。
- `ended_at`：会话结束时间。
- `last_active_at`：最后活跃时间。
- `source_page`：来源页面，例如 `/tools`。
- `message_count`：消息轮数。
- `consent_for_topic`：是否允许匿名加入选题库。
- `audience_group`：可选人群标签。
- `ai_stage`：可选 AI 使用阶段。
- `is_minor_context`：是否为青少年/未成年人场景。
- `prompt_version`：匹配规则版本。
- `model_version`：模型版本。

不保存：

- 原始 IP。
- 姓名、邮箱、手机号、微信。
- API Key。
- 公司/学校精确名称。
- 精确地理位置。

### 4.2 需求事件层：demand_events

保存脱敏后的需求，不保存完整聊天。

字段建议：

- `event_id`
- `session_id`
- `created_at`
- `raw_need_redacted`
- `need_summary`
- `need_category`
- `task_domain`
- `pain_point`
- `misconception`
- `tool_confusion`
- `recommended_content_ids`
- `recommended_tool_ids`
- `clicked_resource_ids`
- `helpfulness`
- `pii_detected`
- `pii_removed`
- `status`：`pending` / `processed` / `ignored`

### 4.3 批处理层：demand_clusters

把相似需求集中处理。

字段建议：

- `cluster_id`
- `date_range`
- `cluster_title`
- `representative_need`
- `need_count`
- `audience_groups`
- `common_pain_points`
- `common_misconceptions`
- `content_gap`
- `priority_score`
- `related_content_ids`
- `status`

### 4.4 选题库层：topic_candidates

只保存适合内容创作的选题，不保存个人身份。

字段建议：

- `topic_id`
- `title`
- `audience`
- `core_question`
- `content_angle`
- `source_cluster_id`
- `source_need_count`
- `related_content_ids`
- `recommended_format`：短视频 / 图文 / 直播 / 长文。
- `priority`
- `status`：待看 / 已采纳 / 已发布 / 暂缓。

## 5. 人群标签策略

### 5.1 可选人群标签

用户可跳过，不强制。

建议选项：

- 学生
- 职场办公
- 内容创作者
- 开发者 / 技术从业者
- 自由职业 / 创业者
- 老师 / 培训者
- 家长
- 青少年 / 未成年
- 其他
- 不想说

### 5.2 AI 使用阶段

比真实年龄更有价值。

建议选项：

- 刚开始
- 会用聊天 AI
- 会用 IDE
- 会用命令行
- 能做完整项目
- 不想说

### 5.3 青少年保护模式

当用户选择“青少年 / 未成年”时：

- 不保存原始问题。
- 不保存完整对话。
- 不做长期画像。
- 不推荐 API Key、中转、命令行危险操作。
- 只进入匿名聚合统计。
- 输出更保守的工具建议。

## 6. 隐私与安全策略

### 6.1 前端提示

输入框附近必须显示：

```text
请不要输入姓名、联系方式、API Key、公司机密等敏感信息。
你的问题可能会被匿名分析，用于改进本站内容和选题库；不会保存你的身份信息。
```

### 6.2 用户控制

必须提供：

- 匿名加入选题库开关。
- 清空对话按钮。
- 不想说选项。
- 敏感信息提醒。

### 6.3 脱敏规则

保存前删除或替换：

- 邮箱。
- 手机号。
- 微信号/QQ 号。
- API Key / token。
- 身份证。
- 精确地址。
- 公司机密表达。
- 过长代码片段。

### 6.4 API Key 财产安全

必须完成：

- API Key 只存在服务端环境变量。
- 前端永远不返回 key。
- 服务端不打印 key。
- 上游错误不完整打印。
- 每 IP 每日限流。
- 每 IP 每分钟限流。
- 全站每日总调用上限。
- 单请求最大字符数。
- 单会话最大轮数。
- 模型调用超时中断。

### 6.5 网站安全

必须完成：

- 前端不直接渲染未转义 HTML。
- 模型输出优先结构化 JSON。
- 链接必须来自站内白名单。
- 推荐资料只允许站内索引 ID。
- 服务端返回 `Cache-Control: no-store`。

## 7. Agent 输出格式

目标是资料包，不是长篇回答。

建议结构：

```text
一句话：
你现在的问题可以按 A → B → C 解决。

推荐资料：
1. 标题：解决哪一步
2. 标题：解决哪一步
3. 标题：解决哪一步

如果看不懂：
把推荐文章或资料原文丢给 AI，让它用生活里的例子讲一遍。
```

约束：

- 最多 5 条资料。
- 优先站内文章和资源。
- 不外部搜索。
- 不输出未入库工具。
- 不长篇解释。
- 不直接替代原文。

## 8. 定时批处理策略

### 8.1 处理频率

初期建议：

- 每天凌晨 2 点处理一次。
- 需求量大后改成每天 2 次。
- 不要每次会话结束就深度分析。

### 8.2 批处理步骤

1. 读取 `pending` 状态的匿名需求。
2. 执行二次脱敏。
3. 去重。
4. 按需求类别初分组。
5. 聚类相似问题。
6. 判断站内内容是否已覆盖。
7. 找出内容缺口。
8. 生成选题候选。
9. 标记已处理。

### 8.3 成本控制

- 批量处理，每次 20-50 条。
- 先用规则分类，模型只处理模糊问题。
- 输入模型的是脱敏摘要，不是完整对话。
- 失败后可重试，但最多 3 次。

## 10. 验收标准

### P0 安全验收

- 输入 HTML 或脚本时，不会在页面执行。
- 直接 POST 超长消息时，接口拒绝且不调用模型。
- 直接 POST 超多轮消息时，接口拒绝或截断且不调用模型。
- API Key 不出现在前端源码、响应体、日志中。
- 上游错误不输出完整正文到日志。
- 用户能看到敏感信息提醒。
- 用户能关闭匿名加入选题库。

### P1 体验验收

- 用户输入一个需求后，返回 3-5 条站内资料。
- 每条资料有标题、链接、用途一句话。
- 输出包含一句串联语。
- 输出包含“看不懂就让 AI 用生活例子解释”的兜底提示。
- 推荐资料全部来自站内索引。
- 不出现外部搜索结果。

### P2 数据验收

- 每次会话记录开始时间、结束时间、消息轮数。
- 选题库只保存脱敏需求，不保存完整对话。
- 需求事件可标记 `pending` / `processed` / `ignored`。
- 每日批处理能把需求聚类成候选选题。
- 青少年/未成年场景不保存原始问题。

### P3 运营验收

- 能查看最近 7 天高频需求。
- 能查看内容缺口。
- 能查看最常见工具误区。
- 能从候选选题进入人工确认。
- 已发布内容能反向关联需求来源。

## 12. 默认决策

遇到分歧时，按以下默认决策执行：

- 安全优先于体验。
- 隐私优先于数据量。
- 站内资料优先于模型生成。
- 匿名需求优先于用户画像。
- 批处理优先于实时深度分析。
- 手动确认优先于自动发布选题。
- 不确定是否敏感时，按敏感处理。
