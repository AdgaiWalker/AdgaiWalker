# Walker 个人网站 PRD

版本：v0.1  
日期：2026-05-26  
状态：初稿  
产品角色：个人认知资产库、对外价值展示站、未来 Agent 可调用知识层

---

## 1. 产品定位

Walker 不是一个人名，而是一种状态：

> 带着目标，准备出发、正在出发、歇会再出发。

Walker 个人网站不是原始灵感箱，也不是普通简历站或工具收藏站。它是一个经过整理和实践验证后的认知资产库：对内帮助 Walker 回看、复用和推进自己的思考；对外展示 Walker 如何用 AI、工具和方法把点子落地为真实结果；长期为 Agent 调用个人知识、行事风格和方法论提供结构化基础。

一句话定位：

> 一个以世界观和方法论为核心，用点子连接人与 AI，把想法摆渡成真实结果的个人网站。

---

## 2. 背景与问题

Walker 的主要思考发生在桌面、Obsidian 和 AI 辅助文档中。那里允许混乱、试错、未验证和反复推翻。网站不承担原始思考发生地的角色，而承担“沉淀层”和“展示层”的角色。

当前需要解决三个问题：

1. 思考容易散落在不同文档中，未来再次前进时难以快速调用。
2. 对外展示容易变成“我是谁、我做了什么”的静态介绍，无法体现底层世界观、方法论和持续实践能力。
3. 点子、工具、知识、项目之间缺少统一结构，难以形成可复用的个人方法系统，也不利于未来 Agent 调用。

因此，第一版网站需要从“因为需要，所以设计”出发：

```text
因为需要前进，所以存在目标。
因为需要落地，所以存在方法。
因为需要验证，所以存在项目和案例。
因为需要减少重复劳动，所以存在工具和自动化。
因为需要再次调用，所以存在知识沉淀。
因为需要连接同行者，所以存在公开展示。
```

## 3. 产品目标

### 3.1 对内目标

- 把已经整理、实践、验证过的思考沉淀为可复用资产。
- 在未来做事时，能够快速调用过去的方法、工具、案例和判断。
- 让点子从“脑内想法”进入“可被验证、可被拆解、可被推进”的结构。
- 为未来 Agent 以 Walker 的行事作风辅助工作打基础。

### 3.2 对外目标

- 让访问者快速理解 Walker 的世界观、目标和实践路径。
- 吸引同频同行者：有点子、愿意动手、相信 AI 可以帮助普通人落地想法的人。
- 建立未来雇主、合作方或社区成员的信任：Walker 不只是会用工具，而是有方法、有判断、有实践结果。
- 让内容对别人产生价值，而不是只做个人情绪记录。

### 3.3 非目标

- 第一版不做复杂登录、私密后台或完整 CMS。
- 第一版不把网站变成 Obsidian 替代品。
- 第一版不追求大而全的社区产品。
- 第一版不公开未经整理和验证的原始想法。

## 4. 用户与优先级

用户优先级：

1. Walker 自己：沉淀、复用、辅助前进。
2. 同频同行者：理解、连接、共创。
3. 未来雇主/合作方：判断能力、实践能力、可信度。

### 4.1 Walker 自己

核心需要：

- 我过去验证过什么？
- 哪些方法能继续复用？
- 哪些工具真正帮我减少了重复劳动？
- 哪些点子值得继续推进？
- 我的世界观和方法论是否越来越清晰？

### 4.2 同频同行者

核心需要：

- Walker 在做什么？
- 我为什么要关注或加入？
- 这里能不能帮助我把自己的点子做出来？
- 我能从哪些文章、工具、案例中获得直接价值？

### 4.3 未来雇主/合作方

核心需要：

- 这个人是否有清晰目标和稳定方法？
- 是否真的做出过东西？
- 是否能分析需求、设计方案、拆任务并交付结果？
- 是否具备 AI 时代的人机协作能力？

## 5. 产品原则

### 5.1 因为需要，所以设计

不先设计栏目，再往里面塞内容。先确认真实需要，再让内容、分类和界面生长出来。

### 5.2 网站是沉淀层，不是原始草稿箱

原始思考留在 Obsidian、桌面文档和 AI 对话中。网站只承载经过整理、对实践有帮助、对别人有价值的内容。

### 5.3 先验证，再公开

点子和观点不急于公开。先行动、验证、纠偏，再沉淀为文章、案例、方法或工具说明。这既是质量控制，也是思想保护。

### 5.4 世界观在前，工具在后

AI 工具不是核心，工具只是手段。网站首先要表达 Walker 如何理解世界、AI、人、重复劳动和前进。

### 5.5 人负责目标、判断和责任

AI 可以辅助生成、拆解、执行和整理，但人必须负责目标选择、价值判断和结果承担。

### 5.6 面向未来 Agent 调用

内容不只为人阅读，也要逐步结构化为 AI 可理解、可检索、可调用的个人知识资产。

## 6. 核心叙事

第一版网站采用三问叙事：

```text
为什么做
做什么
怎么做
```

### 6.1 为什么做

表达 Walker 的世界观：

- AI 的意义不只是提效，而是减少重复劳动。
- 人应该把更多时间从重复劳动中释放出来，去追求真正的幸福。
- 点子不是空想，点子需要在行动中验证。
- 普通人也可以借助 AI，把自己的想法做成真实结果。

### 6.2 做什么

表达网站的承载对象：

- 世界观：关于 AI、实践、人、重复劳动和解放的底层判断。
- 方法论：探索环境、需求分析、验证需求、设计方案、拆分任务、锚定目标、达成结果、整理沉淀。
- 工具：实际使用过、能服务行动的 AI 工具、模型分工、工作流、Skill。
- 知识：实践后沉淀出的文章、复盘和认知。
- 项目：真实做出来的产品、网站、社区、方法框架。
- 点子：经过初步验证、可继续推进或可共创的想法。

### 6.3 怎么做

表达 Walker 方法：

```text
探索环境
→ 需求分析
→ 验证需求
→ 设计方案
→ 拆分任务
→ 锚定目标
→ 达成结果
→ 整理沉淀
```

对应 Walker 的前进状态：

```text
准备出发：观察、收集、判断、定目标。
正在出发：行动、验证、拆解、交付结果。
歇会再出发：复盘、整理、沉淀、再次出发。
```

## 7. 信息架构

第一版采用“外层三问叙事 + 内层认知资产库 + Walker 状态语言”的结构。

### 7.1 顶层导航

第一版默认维持当前站点的三页极简结构，避免过早增加页面和维护成本：

```text
首页 /        承载为什么做、做什么、怎么做
工具 /tools  承载工具、Skill、工作流、社群入口
知识 /posts  承载文章、方法、项目复盘、点子沉淀
```

项目和方法先作为首页与知识库中的强模块存在。等内容量和案例量足够后，再扩展为独立页面：

```text
方法 /method
项目 /projects
点子 /ideas
```

### 7.2 首页结构

首页目标：让人快速理解 Walker 状态、Walker 为什么做、正在做什么、如何连接。

模块建议：

1. Hero：Walker 是一种状态
   - 主标题：Walker
   - 副标题：带着目标，准备出发、正在出发、歇会再出发。
   - 一句话说明：用点子连接人与 AI，把想法摆渡成真实结果。

2. 为什么做
   - AI 不是炫技，而是减少重复劳动。
   - 点子不是空想，而是在行动中验证。
   - 最终去到一个没有重复工作的地方。

3. 做什么
   - 认知资产库：世界观、方法论、工具、知识、项目、点子。
   - 说明网站不是草稿箱，而是验证后的沉淀层。

4. 怎么做
   - 展示 Walker 方法流程。
   - 用“准备出发 / 正在出发 / 歇会再出发”包装流程。

5. 当前实践
   - iwalk.pro：个人认知资产库。
   - xyzidea.com：AI 辅助点子工具站。
   - 点子共促：AI 学以致用与落地实践群。
   - Ferry：人机协作与提示词工程框架。

6. 连接入口
   - GitHub
   - Bilibili / 小红书
   - Email
   - 点子共促社群

### 7.3 方法页

目标：沉淀 Walker 方法，让访客理解“如何把点子落地”。

核心模块：

- Walker 方法总览
- 准备出发：探索环境、收集信息、明确目标
- 正在出发：需求分析、验证需求、设计方案、拆分任务、执行
- 歇会再出发：复盘、沉淀、纠偏、形成可复用资产
- 方法案例：用真实项目说明方法如何工作
- 可下载/可复制模板：需求分析模板、项目复盘模板、点子验证模板

### 7.4 工具页

目标：不是展示“工具很多”，而是展示“工具因需要而存在”。

核心模块：

- 社群共创通道
- 主力模型分工
- 工作流装备
- 常用开发工具
- 自定义 Skill
- 省钱与降耗技巧
- 每个工具标注：解决什么需要、适用场景、实践结果、是否仍在使用

### 7.5 知识页

目标：承载验证后的文章、复盘、知识和方法片段。

内容分组建议：

- 世界观：AI、历史唯物主义、人的解放、重复劳动
- 方法论：Ferry、需求分析、行动验证、复盘
- 实践复盘：项目过程、失败经验、迭代记录
- AI 普及：帮助普通人把点子做出来
- 点子沉淀：已验证或可共创的 idea

### 7.6 项目模块

目标：对外证明 Walker 方法能交付真实结果。

项目卡片字段：

- 项目名称
- 当前状态
- 解决的问题
- 使用的方法
- 使用的工具
- 产出结果
- 复盘链接
- 外部链接

第一版重点项目：

- iwalk.pro：个人认知资产库
- xyzidea.com：AI 辅助点子工具站
- Ferry：人机协作方法框架
- 点子共促：AI 实践与共创社群

## 8. 内容模型

为未来 Agent 调用做准备，每类内容都需要尽量结构化。

### 8.1 文章/知识

字段建议：

```text
title
date
summary
category: world-view | method | practice | ai-literacy | idea
tags
status: thinking | practicing | verified | archived
need: 这篇内容解决什么需要
practice_value: 对 Walker 自己的实践价值
public_value: 对别人有什么价值
related_tools
related_projects
related_ideas
agent_notes: 给未来 Agent 的调用提示
published
```

### 8.2 工具

字段建议：

```text
name
description
category: model | workflow | dev-tool | skill | community | resource
need: 为什么需要它
scenario: 适合什么场景
usage: Walker 如何使用它
result: 带来了什么结果
cost
rating
status: active | testing | archived
url
published
```

### 8.3 项目

字段建议：

```text
name
summary
status: idea | validating | building | launched | archived
problem
target_user
method_steps
tools
result

```

### 8.4 点子

字段建议：

```text

related_tools

published
```

## 9. 核心用户路径

### 9.1 Walker 自用路径

```text
在 Obsidian / AI 文档中产生和整理思考
→ 通过实践验证
→ 判断是否对自己实践有帮助、对别人有价值
→ 以文章、工具、项目或点子形式沉淀到网站
→ 未来做事时通过搜索、标签、关联内容或 Agent 调用
```

### 9.2 同频同行者路径

```text
进入首页
→ 理解 Walker 是一种前进状态
→ 看懂为什么做、做什么、怎么做
→ 阅读方法或案例
→ 进入工具页或知识页获得价值
→ 加入点子共促或联系 Walker
```

### 9.3 雇主/合作方路径

```text
进入首页
→ 快速理解目标和价值观
→ 查看项目与实践结果
→ 查看方法论与复盘文章
→ 判断 Walker 的需求分析、AI 协作、执行和沉淀能力
→ 通过 GitHub / Email 联系
```

### 9.4 未来 Agent 路径

```text
读取结构化内容
→ 识别 Walker 的世界观、方法、工具偏好和项目上下文
→ 根据任务调用相关知识
→ 按 Walker 的行事风格生成建议、拆解任务或整理复盘
```

## 11. 成功指标

### 11.1 对内指标

- 每个月至少沉淀 2-4 篇经过验证的内容。
- 每个重要项目都有对应复盘。
- 工具库中每个工具都有明确使用场景和实践结果。
- 未来做新项目时，能从旧内容中复用方法、模板或判断。

### 11.2 对外指标

- 访问者能在 1 分钟内说清 Walker 在做什么。
- 同频者能理解如何加入或联系。
- 合作方能看到至少 3 个真实项目和对应方法。
- 内容不是“收藏型”，而是能帮助别人行动。

### 11.3 Agent 准备度指标

- 核心内容具备结构化字段。
- 方法论、工具、项目之间有关联。
- 存在一份 Agent 可读的 Walker 风格说明。
- 未来可以基于内容生成任务拆解、项目复盘或行动建议。

## 12. 路线图

### 12.1 0-1 个月：主站重构期

目标：让 iwalk.pro 从个人介绍站升级为 Walker 认知资产库。

重点：

- 首页改为三问叙事。
- 明确 Walker 状态和 Walker 方法。
- 工具页从工具展示改为需要驱动的工具库。
- 知识页按世界观、方法论、实践复盘组织。
- 补齐项目模块的基础展示。

### 12.2 1-3 个月：内容沉淀期

目标：建立稳定的内容筛选和发布机制。

重点：

- 从 Obsidian 和 AI 文档中筛选已验证内容。
- 持续发布世界观、方法论、实践复盘文章。
- 为每个工具补充“需要、场景、结果”。
- 为每个项目补充复盘链接。

### 12.3 3-6 个月：方法验证期

目标：用案例证明 Walker 方法有效。

重点：

- 沉淀 3-5 个点子落地案例。
- 形成点子验证模板、需求分析模板、项目复盘模板。
- 将点子共促社群中的真实问题反哺为内容。
- 逐步完善 Agent 可读字段和风格说明。

### 12.4 6-12 个月：Agent 与社区准备期

目标：让网站成为 Agent 和社区产品的知识底座。

重点：

- 建立结构化知识导出。
- 增加 Agent 调用说明。
- 根据真实内容量决定是否独立点子库和项目页。
- 为未来 xyzidea.cn 社区产品提供案例、方法和工具基础。

## 13. 风险与约束

### 13.1 风险：内容过早公开

应对：坚持“先整理、再验证、后公开”。未验证内容留在 Obsidian。

### 13.2 风险：栏目过多导致维护成本上升

应对：第一版保留少量顶层页面，项目和点子先作为模块沉淀。

### 13.3 风险：工具库变成收藏夹

应对：每个工具必须说明它解决什么需要，以及 Walker 如何使用它。

### 13.4 风险：世界观表达过重，普通访客看不懂

应对：首页用“为什么做、做什么、怎么做”解释，深层文章再展开理论。

### 13.5 风险：Agent 层过早复杂化

应对：第一版只做结构化字段和风格说明，不做在线 Agent 产品。

## 14. 第一版验收标准

- 首页能完整回答：为什么做、做什么、怎么做。
- 首页明确表达：Walker 是一种状态，不只是一个人名。
- 网站内容结构能承载：世界观、方法论、工具、知识、项目、点子。
- 工具页不再只是工具列表，而是能说明工具背后的需要。
- 知识页能区分世界观、方法论、实践复盘等内容。
- 项目展示能体现问题、方法、结果，而不是只放链接。
- 新增内容字段能支持未来 Agent 调用。
- 整体表达对内有复用价值，对外有连接和信任价值。

## 15. 第一版默认决策

1. 第一版维持当前三页结构：首页、工具、知识；方法、项目和点子先作为模块存在。
2. “Walker 方法”作为正式公开命名，同时在文案中解释 Walker 是一种前进状态，不是单一人名。
3. 第一版不新增独立项目页，先在首页展示核心项目，并在知识页沉淀项目复盘。
4. Agent 层第一版先通过内容结构化字段预留能力；后续单独补充 `walker-style.md`，记录 Walker 的世界观、表达风格、做事原则和常用流程。
5. 点子库第一版不公开原始点子，只公开已验证点子、实践案例和适合共创的阶段性成果。
